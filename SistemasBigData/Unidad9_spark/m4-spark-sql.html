1. Catalyst: el optimizador de consultas

Cuando escribes df.filter(...).groupBy(...).agg(...), no se ejecuta tal cual. El optimizador Catalyst convierte tu consulta en 4 planes antes de ejecutar nada:

Plan¿Qué hace?
1. Lógico no resueltoEl AST de tu consulta — columnas sin verificar
2. Lógico analizadoResuelve nombres de columnas, tipos, tablas
3. Lógico optimizadoPredicate pushdown, column pruning, constant folding, join reordering…
4. FísicoElige algoritmos concretos: SortMergeJoin vs BroadcastHashJoin, etc.
python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, avg, count

spark = SparkSession.builder.master("local[*]").appName("m4-sql") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

df = spark.read.parquet("/datos/ventas_2024/")

plan_query = (df
    .filter(col("pais") == "ES")
    .select("producto", "importe", "categoria")
    .groupBy("categoria")
    .agg(avg("importe").alias("ticket_medio"), count("*").alias("n_ventas"))
)

# Ver todos los planes
plan_query.explain()          # solo plan físico
plan_query.explain("simple")  # físico simple
plan_query.explain("extended") # los 4 planes

# plan físico típico — Catalyst ha empujado el filtro ANTES del groupBy:
# == Physical Plan ==
# *(2) HashAggregate(keys=[categoria#5], ...)
# +- Exchange hashpartitioning(categoria#5, ...)
#    +- *(1) HashAggregate(keys=[categoria#5], ...)
#       +- *(1) Project [producto#3, importe#4, categoria#5]
#          +- *(1) Filter (isnotnull(pais#6) AND (pais#6 = ES))  ← pushdown
#             +- *(1) ColumnarToRow
#                +- FileScan parquet [...]

2. Spark SQL: texto SQL sobre DataFrames

python
datos = [
    {"id": 1, "producto": "Laptop", "categoria": "Tech",    "importe": 899.0,  "pais": "ES"},
    {"id": 2, "producto": "Teclado", "categoria": "Tech",   "importe": 79.0,   "pais": "ES"},
    {"id": 3, "producto": "Silla",   "categoria": "Oficina","importe": 350.0,  "pais": "FR"},
    {"id": 4, "producto": "Monitor", "categoria": "Tech",   "importe": 450.0,  "pais": "ES"},
    {"id": 5, "producto": "Mesa",    "categoria": "Oficina","importe": 280.0,  "pais": "DE"},
]
df_ventas = spark.createDataFrame(datos)

# Registrar como vista SQL temporal (en memoria de la sesión Spark)
df_ventas.createOrReplaceTempView("ventas")

# Consultar con SQL puro
resultado = spark.sql("""
    SELECT
        categoria,
        COUNT(*) AS n,
        ROUND(AVG(importe), 2) AS ticket_medio,
        MAX(importe) AS max_venta
    FROM ventas
    WHERE pais = 'ES'
    GROUP BY categoria
    ORDER BY ticket_medio DESC
""")
resultado.show()

# Vista global (persiste entre sesiones de la misma aplicación)
df_ventas.createOrReplaceGlobalTempView("ventas_global")
spark.sql("SELECT * FROM global_temp.ventas_global WHERE importe > 200").show()

# SQL y DataFrame API producen el mismo plan — 100% equivalentes
# SQL:  spark.sql("SELECT producto, importe FROM ventas WHERE pais='ES'")
# API:  df_ventas.filter(col("pais")=="ES").select("producto","importe")

2.1 SQL avanzado: CTEs y subqueries

python
# CTEs (Common Table Expressions)
spark.sql("""
    WITH ventas_es AS (
        SELECT * FROM ventas WHERE pais = 'ES'
    ),
    por_categoria AS (
        SELECT categoria,
               ROUND(AVG(importe), 2) AS media,
               COUNT(*) AS n
        FROM ventas_es
        GROUP BY categoria
    )
    SELECT *,
           RANK() OVER (ORDER BY media DESC) AS rank_cat
    FROM por_categoria
""").show()

# Subquery escalar
spark.sql("""
    SELECT *,
           importe - (SELECT AVG(importe) FROM ventas) AS diff_media
    FROM ventas
    WHERE pais = 'ES'
""").show()

# Funciones de ventana en SQL
spark.sql("""
    SELECT
        producto, importe, pais,
        ROW_NUMBER() OVER (PARTITION BY pais ORDER BY importe DESC) AS rank_pais,
        LAG(importe, 1) OVER (PARTITION BY pais ORDER BY importe DESC) AS importe_anterior
    FROM ventas
""").show()

3. Lectura y escritura de datos

3.1 Parquet — el formato estándar de Spark

ℹ️
¿Por qué Parquet? Almacenamiento columnar, compresión nativa (Snappy/Zstd), schema embebido, predicate pushdown, column pruning. En benchmarks, leer 10 columnas de 100 con Parquet puede ser 30× más rápido que con CSV.
python
# Escribir Parquet
df_ventas.write.mode("overwrite").parquet("/tmp/ventas.parquet")

# Particionado por columna (crea subdirectorios pais=ES/, pais=FR/...)
df_ventas.write \
    .mode("overwrite") \
    .partitionBy("pais", "categoria") \
    .parquet("/tmp/ventas_part/")

# Leer — Catalyst empuja el filtro al nivel de fichero (solo lee partición ES)
df_leido = spark.read.parquet("/tmp/ventas_part/").filter(col("pais") == "ES")
df_leido.explain()  # PushedFilters: [IsNotNull(pais), EqualTo(pais,ES)]

# Opciones avanzadas de escritura
df_ventas.write \
    .mode("append") \          # overwrite, append, ignore, error
    .option("compression", "snappy") \  # snappy, gzip, zstd, none
    .parquet("/tmp/ventas_snappy/")

SortMergeJoinAmbos DF grandes, se ordenan y mezclanAlto — shuffle de red
ShuffleHashJoinUn DF cabe en memoria por particiónMedio

5. Tungsten: motor de ejecución de bajo nivel

Tungsten es la capa que hace que DataFrames sean hardware-efficient: gestión manual de memoria (off-heap), generación de bytecode JVM en tiempo de ejecución (Whole-stage code gen) y procesamiento vectorizado (Arrow).

python
# Configuraciones Tungsten relevantes
spark.conf.set("spark.sql.codegen.wholeStage", "true")              # code gen (defecto true)
spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true") # Apache Arrow para toPandas/fromPandas

# Arrow acelera drásticamente la conversión Spark ↔ Pandas
import pandas as pd
spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true")

# 10× más rápido con Arrow activado
pandas_df = df_ventas.toPandas()
df_desde_pandas = spark.createDataFrame(pandas_df)

# Pandas UDFs (vectorizadas con Arrow — mucho más rápidas que UDFs Python normales)
from pyspark.sql.functions import pandas_udf
from pyspark.sql.types import DoubleType
import pandas as pd

@pandas_udf(DoubleType())
def calcular_neto(s: pd.Series) -> pd.Series:
    """UDF vectorizada — opera sobre Series de pandas, no fila a fila."""
    return s * 0.78

df_ventas.withColumn("importe_neto", calcular_neto(col("importe"))).show()

6. Shuffle, particionado y número de particiones

⚠️
El shuffle es el mayor coste de rendimiento en Spark. Ocurre en: groupBy, join (SortMerge), repartition, distinct, orderBy a gran escala. Cada shuffle implica escribir datos a disco y leerlos en red.
python
# Número de particiones de shuffle (defecto: 200)
# Si tienes pocas filas, 200 particiones son ineficientes (overhead)
spark.conf.set("spark.sql.shuffle.partitions", "50")  # ajustar a tu cluster

# Ver cuántas particiones tiene un DF
print(df_ventas.rdd.getNumPartitions())  # 4 (= CPU cores en local mode)

# repartition: SHUFFLE — redistribuye aleatoriamente
df_200 = df_ventas.repartition(200)
df_por_pais = df_ventas.repartition(10, "pais")  # hash-partition por "pais"

# coalesce: SIN shuffle — solo reduce particiones (une particiones existentes)
# Más eficiente que repartition cuando reduces
df_small = df_ventas.coalesce(4)

# Cuándo usar cada uno:
# repartition(n): para aumentar particiones o redistribuir antes de join/groupBy
# repartition(n, col): para hash-partition antes de join repetidos por esa clave
# coalesce(n): solo para REDUCIR particiones (antes de escribir, por ejemplo)
