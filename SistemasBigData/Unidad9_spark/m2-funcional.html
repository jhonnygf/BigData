M2 — Programación Funcional

El paradigma que subyace a Spark. Antes de tocar un DataFrame, necesitas entender cómo pensar en funcional: datos inmutables, transformaciones puras y composición.

1. ¿Qué es la programación funcional?

La Programación Funcional (FP) trata los programas como evaluaciones de funciones matemáticas. El estado mutable y los efectos secundarios son el enemigo. Spark se basa en este paradigma porque en un cluster distribuido, el estado compartido mutable es imposible de coordinar eficientemente.

Paradigma imperativoParadigma funcional
"Haz esto, luego aquello, luego lo otro""El resultado es f(g(datos))"
Variables que cambian de valorValores inmutables (bindings)
Bucles for/while con acumuladoresmap, filter, reduce
Funciones con efectos secundarios (I/O, logs, BD)Funciones puras — misma entrada = misma salida siempre
El orden de ejecución importaLas transformaciones pueden reordenarse y paralelizarse

2. Funciones puras e inmutabilidad

2.1 Función pura: sin efectos secundarios

python
# ❌ Función IMPURA: modifica estado externo
total = 0
def acumular_impuro(x):
    global total
    total += x      # efecto secundario: modifica variable global
    return total

# ✅ Función PURA: sin estado externo, sin efectos secundarios
def sumar_puro(acumulado, x):
    return acumulado + x   # mismo input → siempre mismo output

# Las funciones puras son predecibles, testables y paralelizables
print(sumar_puro(10, 5))   # siempre 15
print(sumar_puro(10, 5))   # siempre 15 — nunca sorpresas

2.2 Inmutabilidad: no modificar lo que ya existe

La inmutabilidad es el principio de que, una vez creado un dato, no se modifica — en su lugar se crea uno nuevo. Esto elimina toda una clase de bugs (efectos secundarios inesperados entre funciones) y es la base del fault-tolerance de Spark.

¿Python da cobertura a esto?

La respuesta corta es: parcialmente, y por convención más que por imposición del compilador. Python es multiparadigma — te deja hacer lo que quieras. Scala, en cambio, fue diseñado desde cero con la programación funcional en mente y tiene mecanismos más fuertes:

ConceptoScala (diseñado para FP)Python (multiparadigma)
Variable inmutable val x = 5 — el compilador prohíbe reasignar x No existe. Solo convención: nombres en UPPER_CASE o Final de typing (no validado en runtime)
Variable mutable var x = 5 — explícitamente mutable Todo es mutable por defecto
Lista inmutable List(1,2,3) — inmutable por defecto. ListBuffer si quieres mutable tuple o frozenset. La list es mutable por defecto
Objeto inmutable case class — todos los campos son val por defecto, .copy() gratis @dataclass(frozen=True) o NamedTuple — aproximación funcional
Verificación compilador ✅ El compilador rechaza mutación de val ❌ Solo herramientas de análisis estático (mypy, pyright) con Final

Lo que Python sí ofrece nativamente

python — tipos inmutables integrados en el lenguaje
# ── 1. Tipos primitivos: siempre inmutables ────────────────────────
# int, float, str, bool, bytes — Python nunca los muta in-place
x = "hola"
y = x.upper()   # crea un string NUEVO, x sigue siendo "hola"
print(x)        # "hola"  ← sin cambios
print(y)        # "HOLA"

# ── 2. tuple — secuencia inmutable ────────────────────────────────
coordenadas = (40.416, -3.703)
# coordenadas[0] = 41.0  → TypeError: 'tuple' object does not support item assignment
nueva_coord = (41.0, coordenadas[1])   # creas una nueva tupla

# ── 3. frozenset — conjunto inmutable ─────────────────────────────
etiquetas = frozenset({"spark", "python", "bigdata"})
# etiquetas.add("scala")  → AttributeError: 'frozenset' object has no attribute 'add'
nuevas = etiquetas | {"scala"}   # crea un frozenset NUEVO

# ── 4. str — siempre immutable (a diferencia de Java StringBuilder)
s = "spark"
# s[0] = "S"  → TypeError
s2 = "S" + s[1:]    # nuevo string

Patrones para inmutabilidad en tus propias clases

python — dataclass frozen y NamedTuple
from dataclasses import dataclass
from typing import NamedTuple, Final

# ── Opción A: dataclass frozen=True ───────────────────────────────
# frozen=True genera __setattr__ y __delattr__ que lanzan FrozenInstanceError
@dataclass(frozen=True)
class Sensor:
    sensor_id: str
    caudal:    float
    presion:   float

s1 = Sensor("sensor-01", 18.4, 2.7)
# s1.caudal = 99.0   → FrozenInstanceError: cannot assign to field 'caudal'

# "Modificar" = crear un nuevo objeto (similar a .copy() de Scala case class)
s2 = dataclass.replace(s1, caudal=22.0)   # nuevo objeto, s1 intacto
print(s1.caudal)   # 18.4
print(s2.caudal)   # 22.0

# ── Opción B: NamedTuple — más ligera, también hasheable ──────────
class Punto(NamedTuple):
    x: float
    y: float

p1 = Punto(1.0, 2.0)
# p1.x = 9   → AttributeError
p2 = p1._replace(x=5.0)   # nuevo objeto

# ── Opción C: Final — señal para el analizador estático ───────────
# No impone inmutabilidad en runtime, pero mypy/pyright la detectan
MAX_WORKERS: Final = 8
# MAX_WORKERS = 16   → mypy: Cannot assign to final name "MAX_WORKERS"

¿Y las listas? El patrón inmutable vs mutable

python — la list es mutable, tuple es inmutable
# ❌ Estilo mutable: modifica in-place → efectos secundarios
salarios = [42000, 35000, 48000]

def aplicar_aumento(lista):
    for i in range(len(lista)):
        lista[i] *= 1.1   # ← MUTA la lista original (efecto secundario)

aplicar_aumento(salarios)
print(salarios)   # [46200, 38500, 52800] ← la función alteró el dato externo

# ✅ Estilo inmutable: siempre crea una colección nueva
salarios = [42000, 35000, 48000]

def con_aumento(lista):
    return [s * 1.1 for s in lista]   # ← nueva lista, la original intacta

salarios_nuevos = con_aumento(salarios)
print(salarios)       # [42000, 35000, 48000]  ← sin cambios
print(salarios_nuevos)  # [46200.0, 38500.0, 52800.0]

# En Spark esto es automático:
# df2 = df.withColumn("salario", df.salario * 1.1)
# df nunca cambia — df2 es un nuevo plan de ejecución

Resumen: Python vs Scala para programación funcional

⚖️
Python no te protege del compilador — puedes ignorar todos estos patrones y mutar datos libremente. Scala te obliga (con val) o al menos lo hace explícito (con var). En Python la inmutabilidad es una disciplina de escritura, no una garantía del lenguaje. Por eso en equipos grandes se usan linters y type checking (mypy) para reforzarla.

La buena noticia: Spark no depende de que Python sea funcional. Los DataFrames son inmutables a nivel de motor (JVM), independientemente de lo que hagas en el lado Python.
ℹ️
¿Por qué importa en Spark? Los RDDs y DataFrames son inmutables por diseño de motor. Cuando haces df2 = df.filter(...), df nunca cambia — df2 es un nuevo plan. Esto permite al scheduler re-ejecutar cualquier partición perdida (fault tolerance) sin riesgo de que el reintento produzca un resultado diferente.

3. Las tres operaciones fundamentales: map, filter, reduce

Estas tres funciones de orden superior son la base de todo procesamiento de datos en Spark. Entenderlas en Python puro hace que la API de Spark sea obvia.

3.1 map — transformar cada elemento

python
empleados = [
    {"nombre": "Alice", "salario": 42000, "dpto": "Ingeniería"},
    {"nombre": "Bob",   "salario": 35000, "dpto": "Marketing"},
    {"nombre": "Clara", "salario": 48000, "dpto": "Ingeniería"},
]

# map aplica una función a CADA elemento y devuelve un iterable nuevo
# No modifica la lista original

# Extraer solo nombres
nombres = list(map(lambda e: e["nombre"], empleados))
print(nombres)   # ['Alice', 'Bob', 'Clara']

# Aplicar aumento del 10%
con_aumento = list(map(lambda e: {**e, "salario": e["salario"] * 1.1}, empleados))
print(con_aumento[0])  # {'nombre': 'Alice', 'salario': 46200.0, 'dpto': 'Ingeniería'}

# Equivalente con comprehension (más idiomático en Python)
nombres_comp = [e["nombre"] for e in empleados]

# En Spark esto es:  df.select("nombre")
# map sobre RDD:     rdd.map(lambda e: e["nombre"])

3.2 filter — seleccionar elementos que cumplen una condición

python
# filter aplica un predicado y devuelve solo los True
ingenieria = list(filter(lambda e: e["dpto"] == "Ingeniería", empleados))
print(len(ingenieria))  # 2

# Encadenar: filtrar + transformar (composición)
salarios_ing = list(map(
    lambda e: e["salario"],
    filter(lambda e: e["dpto"] == "Ingeniería", empleados)
))
print(salarios_ing)  # [42000, 48000]

# Equivalente con comprehension
salarios_ing_comp = [e["salario"] for e in empleados if e["dpto"] == "Ingeniería"]

# En Spark esto es:  df.filter(df.dpto == "Ingeniería").select("salario")

3.3 reduce — colapsar una colección a un valor

python
from functools import reduce

salarios = [42000, 35000, 48000, 39000, 51000]

# reduce(función, iterable, valor_inicial)
# función recibe (acumulado, elemento_actual) → nuevo_acumulado
total = reduce(lambda acc, x: acc + x, salarios, 0)
print(total)   # 215000

maximo = reduce(lambda acc, x: x if x > acc else acc, salarios)
print(maximo)  # 51000

# En Spark:
# rdd.reduce(lambda a, b: a + b)
# df.agg({"salario": "sum"})
# df.agg(F.sum("salario"))

# Visualizando el proceso de reduce:
# acc=0,     x=42000 → acc=42000
# acc=42000, x=35000 → acc=77000
# acc=77000, x=48000 → acc=125000 ... etc.

3.4 Encadenamiento: el pipeline funcional

python
# Problema: salario medio del departamento de Ingeniería con aumento del 5%

# Estilo imperativo (difícil de paralelizar)
ing_salarios = []
for e in empleados:
    if e["dpto"] == "Ingeniería":
        ing_salarios.append(e["salario"] * 1.05)
media = sum(ing_salarios) / len(ing_salarios)

# Estilo funcional (composición de transformaciones puras)
ing_salarios_f = list(map(
    lambda s: s * 1.05,
    map(lambda e: e["salario"],
        filter(lambda e: e["dpto"] == "Ingeniería", empleados))
))
media_f = reduce(lambda a, b: a + b, ing_salarios_f) / len(ing_salarios_f)

# En Spark, este mismo pipeline se escribe:
# (df
#   .filter(col("dpto") == "Ingeniería")
#   .withColumn("salario_proyectado", col("salario") * 1.05)
#   .agg(avg("salario_proyectado"))
# )

4. Funciones lambda

Una lambda es una función anónima de una expresión. En Python: lambda parámetros: expresión. Son la forma natural de pasar funciones a map, filter, reduce y a las API de Spark.

python
# Lambda básica
cuadrado = lambda x: x ** 2
print(cuadrado(5))   # 25

# Lambda con múltiples parámetros
suma = lambda a, b: a + b
print(suma(3, 4))    # 7

# Lambda con condición (ternario)
clasificar = lambda s: "alto" if s >= 45000 else "bajo"
print(clasificar(42000))   # "bajo"
print(clasificar(51000))   # "alto"

# ⚠️ Límite: lambdas solo pueden tener UNA expresión
# Para lógica compleja, usa def — Spark acepta ambos igual
def procesar_empleado(e):
    dpto = e["dpto"].upper()
    salario_neto = e["salario"] * 0.78  # 22% IRPF aprox.
    return {**e, "dpto_norm": dpto, "salario_neto": round(salario_neto, 2)}

procesados = list(map(procesar_empleado, empleados))

# Diferencia lambdas vs def con Spark UDFs
from pyspark.sql.functions import udf
from pyspark.sql.types import DoubleType

# Ambas formas son válidas para registrar como UDF
calcular_neto_lambda = udf(lambda s: round(s * 0.78, 2), DoubleType())
calcular_neto_def    = udf(procesar_empleado, DoubleType())  # si devolviera solo float

5. Funciones de orden superior

Una función que recibe otras funciones como argumento, o que devuelve una función, se llama función de orden superior. map, filter y reduce son ejemplos. Pero puedes crearlas tú mismo.

python
# Función que devuelve una función: "fábrica de funciones"
def crear_filtro_dpto(dpto_objetivo):
    """Devuelve un predicado que filtra por departamento."""
    return lambda empleado: empleado["dpto"] == dpto_objetivo

# Crear predicados especializados
es_ingenieria = crear_filtro_dpto("Ingeniería")
es_marketing  = crear_filtro_dpto("Marketing")

print(es_ingenieria({"dpto": "Ingeniería"}))  # True
print(es_marketing({"dpto": "Ingeniería"}))   # False

ingenieria = list(filter(es_ingenieria, empleados))

# Función que recibe otra función como argumento
def aplicar_a_salarios(empleados, transformacion):
    return [transformacion(e["salario"]) for e in empleados]

bonos = aplicar_a_salarios(empleados, lambda s: s * 0.1)
print(bonos)  # [4200.0, 3500.0, 4800.0]

# Patrón muy común en Spark: pasar funciones a .transform()
def normalizar_columnas(df):
    """Transforma un DataFrame: normaliza nombres de columnas."""
    from pyspark.sql import functions as F
    return df.select([F.col(c).alias(c.lower().replace(" ", "_"))
                      for c in df.columns])

# df_limpio = df.transform(normalizar_columnas)

6. Closures: funciones que "recuerdan" su contexto

Un closure es una función que captura variables de su entorno de definición. Es la base de las "fábricas de funciones". En Spark, esto tiene una trampa importante.

python
# Closure simple
def multiplicador(factor):
    def multiplicar(x):
        return x * factor   # factor es capturado del entorno exterior
    return multiplicar

doble  = multiplicador(2)
triple = multiplicador(3)

print(doble(5))   # 10
print(triple(5))  # 15

# Trampas de closures en bucles (¡error clásico!)
funciones_mal = [lambda x: x * i for i in range(3)]
# Todas capturan la misma variable i (su valor final = 2)
print([f(10) for f in funciones_mal])  # [20, 20, 20] ← ¡mal!

# Solución: capturar el valor actual con argumento por defecto
funciones_bien = [lambda x, i=i: x * i for i in range(3)]
print([f(10) for f in funciones_bien])  # [0, 10, 20] ← correcto
⚠️
Closures en Spark: cuando una función lambda o def que pasas a Spark captura variables del Driver, esas variables se serializan y envían a cada Executor. Si la variable es enorme (un modelo ML, un diccionario de millones de entradas), esto es un cuello de botella. Usa Broadcast variables para datos grandes compartidos.

7. Composición de funciones

Componer funciones significa aplicar varias en secuencia. h = f ∘ g significa "aplica g, luego aplica f al resultado". En Spark, cada transformación del DataFrame es una operación de composición.

python
from functools import reduce as fred

# Definir operaciones como funciones puras
def normalizar_nombre(e): return {**e, "nombre": e["nombre"].strip().title()}
def calcular_neto(e):     return {**e, "neto": round(e["salario"] * 0.78, 2)}
def categorizar(e):       return {**e, "nivel": "senior" if e["salario"] > 45000 else "junior"}

# Componer: aplicar las tres en secuencia
pipeline = [normalizar_nombre, calcular_neto, categorizar]

def aplicar_pipeline(empleado, pipeline):
    return fred(lambda e, f: f(e), pipeline, empleado)

procesado = aplicar_pipeline(
    {"nombre": " alice ", "salario": 42000, "dpto": "Ingeniería"},
    pipeline
)
print(procesado)
# {'nombre': 'Alice', 'salario': 42000, 'dpto': 'Ingeniería', 'neto': 32760.0, 'nivel': 'junior'}

# En Spark, el mismo concepto usando .transform()
def step1_normalizar(df):
    from pyspark.sql.functions import initcap, trim
    return df.withColumn("nombre", initcap(trim(df.nombre)))

def step2_calcular_neto(df):
    from pyspark.sql.functions import round as spark_round
    return df.withColumn("neto", spark_round(df.salario * 0.78, 2))

def step3_categorizar(df):
    from pyspark.sql.functions import when
    return df.withColumn("nivel",
        when(df.salario > 45000, "senior").otherwise("junior"))

# Aplicar pipeline sobre un DataFrame — limpio y refactorizable
# resultado = df.transform(step1_normalizar).transform(step2_calcular_neto).transform(step3_categorizar)

8. Recursión

En FP pura, la recursión sustituye a los bucles. Python tiene límite de recursión (~1000 niveles), por lo que en práctica se usan map/filter/reduce o comprehensions. Pero entender la recursión ayuda a comprender cómo Spark divide el trabajo.

python
# Implementar reduce con recursión (pedagogía)
def mi_reduce(funcion, lista, inicial=None):
    if not lista:
        return inicial
    if inicial is None:
        return mi_reduce(funcion, lista[1:], lista[0])
    return mi_reduce(funcion, lista[1:], funcion(inicial, lista[0]))

print(mi_reduce(lambda a, b: a + b, [1, 2, 3, 4, 5]))  # 15

# Flatten recursivo de listas anidadas
def aplanar(lista):
    if not lista:
        return []
    cab, *resto = lista
    if isinstance(cab, list):
        return aplanar(cab) + aplanar(resto)
    return [cab] + aplanar(resto)

print(aplanar([1, [2, 3], [4, [5, 6]]]))  # [1, 2, 3, 4, 5, 6]
# En Spark: df.select(explode("columna_array"))

E1
Dado el listado de empleados, calcula en una sola expresión (sin variables intermedias, sin bucles) el salario promedio de los empleados con salario mayor a 40.000 €.
Usa solo filter, map, reduce y operadores aritméticos.
▸ Ver solución
python
from functools import reduce

empleados = [
    {"nombre": "Ana",   "salario": 42000},
    {"nombre": "Luis",  "salario": 35000},
    {"nombre": "María", "salario": 51000},
    {"nombre": "Pedro", "salario": 39000},
    {"nombre": "Laura", "salario": 48000},
]

# Una sola expresión: filter → map → reduce para suma, dividir entre count
filtrados = list(filter(lambda e: e["salario"] > 40_000, empleados))

promedio = (
    reduce(lambda acc, s: acc + s,
           map(lambda e: e["salario"], filtrados))
    / len(filtrados)
)

print(promedio)   # 47000.0

# ── Versión REALMENTE sin variables intermedias (una expr) ────────
promedio2 = (
    lambda emp: (
        reduce(lambda a, s: a + s, map(lambda e: e["salario"], emp)) / len(emp)
    )
)(list(filter(lambda e: e["salario"] > 40_000, empleados)))

print(promedio2)  # 47000.0
E2
Crea una fábrica de funciones crear_transformador_salario(porcentaje, impuesto) que devuelva una función que aplique primero el porcentaje de aumento y luego descuente el impuesto.
Úsala con map sobre la lista de empleados.
▸ Ver solución
python
def crear_transformador_salario(porcentaje, impuesto):
    """Fábrica de funciones: cierre sobre porcentaje e impuesto."""
    def transformar(empleado):
        salario_bruto   = empleado["salario"] * (1 + porcentaje / 100)
        salario_neto    = salario_bruto * (1 - impuesto / 100)
        return {**empleado, "salario": round(salario_neto, 2)}
    return transformar

empleados = [
    {"nombre": "Ana",   "salario": 42000},
    {"nombre": "Luis",  "salario": 35000},
    {"nombre": "María", "salario": 51000},
]

# Aumento del 10%, IRPF del 21%
transformador = crear_transformador_salario(porcentaje=10, impuesto=21)

resultado = list(map(transformador, empleados))
for e in resultado:
    print(f"{e['nombre']:10} → {e['salario']:>10,.2f} €")

# Ana        →  36,498.00 €
# Luis       →  30,415.00 €
# María      →  44,373.90 €
E3
Implementa group_by(iterable, clave) que agrupe una lista de dicts por el valor de una clave, usando solo funciones puras y reduce. Sin usar itertools.groupby.
▸ Ver solución
python
from functools import reduce

def group_by(iterable, clave):
    """Agrupa una lista de dicts por el valor de 'clave' usando reduce.
    Funciones puras: no muta nada fuera de la función.
    """
    def acumular(grupos, elemento):
        k = elemento[clave]
        # dict unpacking para NO mutar el acumulador anterior
        grupo_actual = grupos.get(k, [])
        return {**grupos, k: grupo_actual + [elemento]}

    return reduce(acumular, iterable, {})

empleados = [
    {"nombre": "Ana",   "dept": "Ingeniería", "salario": 42000},
    {"nombre": "Luis",  "dept": "Marketing",  "salario": 35000},
    {"nombre": "María", "dept": "Ingeniería", "salario": 51000},
    {"nombre": "Pedro", "dept": "Marketing",  "salario": 39000},
    {"nombre": "Laura", "dept": "Ingeniería", "salario": 48000},
]

resultado = group_by(empleados, "dept")

for dept, emps in resultado.items():
    nombres = [e["nombre"] for e in emps]
    print(f"{dept}: {nombres}")

# Ingeniería: ['Ana', 'María', 'Laura']
# Marketing:  ['Luis', 'Pedro']
