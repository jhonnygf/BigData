M6 — Structured Streaming

Procesamiento de datos en tiempo real. Structured Streaming trata un stream como una tabla que crece infinitamente — las mismas transformaciones DataFrame/SQL, ejecutadas continuamente.

1. El modelo mental: tabla que crece

La gran idea de Structured Streaming es que no inventaron una nueva API. Un stream es un DataFrame cuyas filas llegan continuamente y Spark ejecuta repetidamente tu lógica sobre los nuevos datos.

ConceptoBatchStreaming
Fuente de datosFichero CSV, tabla DB completaSocket, Kafka, directorio que crece
Cuándo se ejecutaUna vez, al lanzar el jobContinuamente — cada micro-batch
APIspark.readspark.readStream
Escrituradf.writedf.writeStream
TransformacionesIdénticasIdénticas (con algunas restricciones)
ℹ️
Internamente, Structured Streaming divide el tiempo en micro-batches. En cada trigger, lee los nuevos datos, ejecuta el plan Catalyst como un job batch normal, y hace commit del resultado. El estado persistente entre micro-batches se guarda en un checkpoint.

2. Sources — Cómo llegan los datos

2.1 Socket TCP (para demos)

python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("DemoSocket") \
    .getOrCreate()

# Lee líneas de texto desde un socket TCP
# nc -lk 9999  ← en otra terminal para simular datos
lines = spark.readStream \
    .format("socket") \
    .option("host", "localhost") \
    .option("port", 9999) \
    .load()

# lines es un DataFrame con una columna: "value" (string)
words = lines.selectExpr("explode(split(value, ' ')) AS word")
counts = words.groupBy("word").count()

query = counts.writeStream \
    .outputMode("complete") \
    .format("console") \
    .start()

query.awaitTermination()  # bloquea hasta que se detenga
⚠️
Socket solo para demos. No tiene tolerancia a fallos — si el job se cae, los datos recibidos se pierden. Para producción usa Kafka o ficheros.

2.2 Directorio de ficheros (auto-ingest)

python
from pyspark.sql.types import StructType, StructField, StringType, DoubleType, TimestampType

schema = StructType([
    StructField("timestamp",  TimestampType()),
    StructField("sensor_id",  StringType()),
    StructField("caudal_lpm", DoubleType()),
    StructField("presion_bar", DoubleType()),
])

# Spark detecta automáticamente ficheros nuevos que llegan a /datos/entrada/
df = spark.readStream \
    .schema(schema) \
    .option("maxFilesPerTrigger", 10) \  # cuántos ficheros por micro-batch
    .json("/datos/entrada/")

df.writeStream \
    .format("parquet") \
    .option("path", "/datos/salida/") \
    .option("checkpointLocation", "/checkpoint/ingest/") \
    .start()

2.3 Kafka (producción)

python
import json
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import StructType, StructField, StringType, DoubleType, LongType

# Schema del JSON que llega en Kafka
sensor_schema = StructType([
    StructField("sensor_id",   StringType()),
    StructField("timestamp",   LongType()),    # epoch ms
    StructField("caudal_lpm",  DoubleType()),
    StructField("presion_bar", DoubleType()),
    StructField("temp_c",      DoubleType()),
])

raw = spark.readStream \
    .format("kafka") \

    .option("subscribe", "sensores-agua") \
    .option("startingOffsets", "latest") \
    .load()

# raw tiene columnas: key, value(bytes), topic, partition, offset, timestamp
# Deserializamos el JSON del campo value
parsed = raw \

parsed.printSchema()

#  |-- temp_c:      double (nullable = true)

3. Output Modes — ¿Qué escribir en cada trigger?

Modo¿Qué escribe?Cuándo usarloCompatible con
appendSolo las filas NUEVAS del batch actualSin agregaciones, o con watermarkSelect, filter, map, join con watermark
completeLa tabla de resultados COMPLETA en cada triggerAgregaciones sin watermark (conteos globales)groupBy + agg
updateSolo las filas que han CAMBIADO desde el último triggerAgregaciones con o sin watermarkgroupBy + agg
python — Ejemplo práctico de los 3 modos
from pyspark.sql.functions import count, avg

# Conteo de eventos por sensor
conteos = parsed.groupBy("sensor_id").agg(
    count("*").alias("total_lecturas"),
    avg("caudal_lpm").alias("caudal_medio")
)

# MODO COMPLETE: escribe toda la tabla en cada trigger
q1 = conteos.writeStream \
    .outputMode("complete") \
    .format("console") \
    .option("truncate", False) \
    .start()

# MODO UPDATE: solo los sensores cuyo conteo cambió
q2 = conteos.writeStream \
    .outputMode("update") \
    .format("memory") \
    .queryName("stats_sensores") \
    .start()

# Consultar la tabla en memory desde otra celda/proceso
spark.sql("SELECT * FROM stats_sensores ORDER BY total_lecturas DESC").show()

4. Triggers — ¿Con qué frecuencia procesar?

python
from pyspark.sql.streaming import DataStreamWriter

# 1. Default: procesa el siguiente batch en cuanto termina el anterior
df.writeStream.trigger()

# 2. Fixed interval: procesa cada N segundos/minutos
df.writeStream.trigger(processingTime="10 seconds")
df.writeStream.trigger(processingTime="1 minute")

# 3. Once: un único batch y para (útil para testing y migraciones)
df.writeStream.trigger(once=True)

# 4. Available Now: como Once pero procesa TODOS los datos disponibles
#    (múltiples batches si hay mucha acumulación)
df.writeStream.trigger(availableNow=True)

# 5. Continuous (experimental): latencia ~1ms, sin micro-batches
#    Solo soporta select/filter/map — sin agregaciones
df.writeStream.trigger(continuous="1 second")
💡
Para la mayoría de casos de IoT y monitorización, processingTime="5 seconds" a "30 seconds" ofrece un buen balance entre latencia y overhead. Bajarlo mucho aumenta el overhead del scheduler sin reducir la latencia del negocio proporcionalmente.

5. Ventanas Temporales

Las ventanas son la forma de agrupar datos por tiempo en streaming. Se aplican sobre una columna de tipo timestamp.

5.1 Tumbling window (ventana fija sin solapamiento)

python
from pyspark.sql.functions import window, avg, max, min
from pyspark.sql.functions import to_timestamp, col

# Convertir epoch ms a timestamp si es necesario
df_ts = parsed.withColumn(
    "event_time",
    to_timestamp(col("timestamp") / 1000)  # epoch ms → timestamp
)

# TUMBLING WINDOW: ventanas de 1 minuto sin solapamiento
stats_1min = df_ts \
    .groupBy(
        window(col("event_time"), "1 minute"),  # ventana de 1 min
        col("sensor_id")
    ).agg(
        avg("caudal_lpm").alias("caudal_medio"),
        max("presion_bar").alias("pres_max"),
        min("presion_bar").alias("pres_min")
    )

stats_1min.writeStream \
    .outputMode("append") \
    .format("console") \
    .option("truncate", False) \
    .start()

5.2 Sliding window (ventanas con solapamiento)

python
# SLIDING WINDOW: ventana de 5 minutos, avanza cada 1 minuto
# Un evento pertenece a 5 ventanas distintas simultáneamente
stats_sliding = df_ts \
    .groupBy(
        window(col("event_time"), "5 minutes", "1 minute"),
        col("sensor_id")
    ).agg(
        avg("caudal_lpm").alias("caudal_medio_5m")
    )

# Salida: la ventana [09:00, 09:05) se actualiza cada minuto
# con los nuevos datos que entran en esa ventana

6. Watermarks — Datos que llegan tarde

En sistemas reales los datos llegan desviados del orden temporal: un sensor puede mandar una lectura de las 09:00 cuando el sistema ya va por las 09:10. Sin watermark, Spark mantiene todas las ventanas abiertas indefinidamente consumiendo RAM. Con watermark le dices: "espera máximo N minutos de retraso; después cierra la ventana".

python
from pyspark.sql.functions import window, avg, col

# withWatermark DEBE ir antes del groupBy
df_with_watermark = df_ts \
    .withWatermark("event_time", "2 minutes")  # acepta hasta 2 min de retraso

stats_watermarked = df_with_watermark \
    .groupBy(
        window(col("event_time"), "1 minute"),
        col("sensor_id")
    ).agg(avg("caudal_lpm").alias("caudal_1m"))

# Con watermark podemos usar outputMode="append":
# Spark garantiza que una ventana cerrada no cambiará
stats_watermarked.writeStream \
    .outputMode("append") \       # ahora posible con watermark
    .option("checkpointLocation", "/checkpoint/stats/") \
    .format("parquet") \
    .option("path", "/resultados/stats_1min/") \
    .start()
ℹ️
Watermark vs trigger: el watermark define cuánto tiempo esperas datos tardíos (latencia de evento). El trigger define con qué frecuencia Spark procesa (latencia de procesamiento). Son ortogonales — puedes tener trigger cada 10s y watermark de 5 minutos.

7. Checkpointing y Tolerancia a Fallos

El checkpoint guarda el estado de la query (offsets procesados, estado de ventanas) en almacenamiento persistente. Si el job cae, se reinicia exactamente donde lo dejó — exactly-once semantics.

python
query = df.writeStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("topic", "resultados-alertas") \
    .option("checkpointLocation", "/checkpoint/alertas/") \
    .outputMode("append") \
    .trigger(processingTime="5 seconds") \
    .start()

# Monitorización de la query en curso
print(f"Query ID: {query.id}")
print(f"Status:   {query.status}")
print(f"Progress: {query.lastProgress}")

# Parar limpiamente
query.stop()
¿Qué guarda el checkpoint?Ubicación
Offsets de Kafka procesadoscheckpointLocation/offsets/
Estado de ventanas / stateful opscheckpointLocation/state/
Commits confirmados (exactly-once)checkpointLocation/commits/

8. Operaciones con Estado (Stateful)

Más allá de las agregaciones por ventana, puedes mantener estado arbitrario por clave con mapGroupsWithState / flatMapGroupsWithState. Esto permite detectar patrones complejos que cruzan múltiples micro-batches.

python — Detector de sensor inactivo
from pyspark.sql.streaming.state import GroupState, GroupStateTimeout
from typing import Iterator, Tuple
import datetime

# Estado por sensor: última vez que recibimos datos
class SensorState:
    def __init__(self, last_seen: datetime.datetime, alerted: bool):
        self.last_seen = last_seen
        self.alerted   = alerted

def update_sensor(
    sensor_id: str,
    inputs: Iterator,
    state: GroupState
) -> Iterator:
    MAX_SILENCE = datetime.timedelta(seconds=30)  # si no da señal en 30s → alerta

    now = datetime.datetime.now()
    rows = list(inputs)

    if state.hasTimedOut:
        # No llegaron datos en el timeout configurado
        yield (sensor_id, "INACTIVO", now)
        state.remove()
        return

    # Actualizar última vez visto
    st = state.get if state.exists else SensorState(now, False)
    st.last_seen = now
    st.alerted   = False
    state.update(st)

    # Programar timeout: si en 30s no llegan datos, lanzar alerta
    state.setTimeoutDuration("30 seconds")

    for row in rows:
        yield (sensor_id, "OK", row.event_time)

# Aplicar a un stream agrupado por sensor_id
from pyspark.sql.functions import col

result = df_ts \
    .groupBy("sensor_id") \
    .applyInPandasWithState(
        update_sensor,
        outputStructType="sensor_id STRING, estado STRING, ts TIMESTAMP",
        stateStructType="last_seen TIMESTAMP, alerted BOOLEAN",
        outputMode="append",
        timeoutConf=GroupStateTimeout.ProcessingTimeTimeout
    )

result.writeStream.format("console").outputMode("append").start()

9. Pipeline Kafka completo — ejemplo end-to-end

Lectura de Kafka → detección de anomalía simple → escritura de alertas de vuelta a Kafka:

python — Pipeline IoT completo
from pyspark.sql import SparkSession
from pyspark.sql.functions import (
    from_json, col, to_json, struct, lit,
    when, to_timestamp
)
from pyspark.sql.types import *

spark = SparkSession.builder \
    .appName("MonitorAgua") \
    .config("spark.jars.packages",
            "org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0") \
    .getOrCreate()

spark.sparkContext.setLogLevel("WARN")

# ── Schema ─────────────────────────────────────────
schema = StructType([
    StructField("sensor_id",   StringType()),
    StructField("ts_epoch_ms", LongType()),
    StructField("caudal_lpm",  DoubleType()),
    StructField("presion_bar", DoubleType()),
    StructField("temp_c",      DoubleType()),
])

# ── Lectura de Kafka ────────────────────────────────
raw = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("subscribe", "sensores-agua") \
    .option("startingOffsets", "latest") \
    .load() \
    .selectExpr("CAST(value AS STRING) AS json_str") \
    .select(from_json("json_str", schema).alias("d")) \
    .select("d.*") \
    .withColumn("event_time",
                to_timestamp(col("ts_epoch_ms") / 1000))

# ── Lógica de negocio: detección básica ────────────
# Caudal > 50 lpm O presión < 0.5 bar → posible fuga
with_alert = raw.withColumn(
    "alerta",
    when(
        (col("caudal_lpm") > 50) | (col("presion_bar") < 0.5),
        lit("FUGA_POSIBLE")
    ).otherwise(lit("OK"))
)

# Sólo emitir alertas reales a Kafka
alertas = with_alert.filter(col("alerta") == "FUGA_POSIBLE") \
    .select(
        col("sensor_id").alias("key"),
        to_json(struct(
            "sensor_id", "event_time",
            "caudal_lpm", "presion_bar", "alerta"
        )).alias("value")
    )

# ── Escritura a Kafka (topic alertas) ──────────────
query = alertas.writeStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("topic", "alertas-fugas") \
    .option("checkpointLocation", "/tmp/checkpoint/alertas/") \
    .outputMode("append") \
    .trigger(processingTime="5 seconds") \
    .start()

query.awaitTermination()
