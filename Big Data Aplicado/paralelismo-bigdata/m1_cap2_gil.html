El GIL: el enemigo que debes conocer

El Global Interpreter Lock es la razón por la que Python con múltiples hilos no escala en CPU-bound. Entenderlo evita decisiones de diseño incorrectas.

Qué es el GIL y por qué existe

El GIL (Global Interpreter Lock) es un mutex — un bloqueo de exclusión mutua — que protege el intérprete de CPython. En cualquier momento dado, solo un hilo puede ejecutar bytecode Python. Aunque tengas 16 hilos y 16 núcleos de CPU, el intérprete de Python solo ejecuta uno a la vez.

Existe desde los primeros días de Python por una razón concreta: CPython gestiona la memoria con un sistema de conteo de referencias. Cada objeto Python tiene un contador interno que registra cuántos otros objetos lo apuntan. Cuando ese contador llega a cero, el objeto se libera. Sin protección, dos hilos modificando el contador del mismo objeto al mismo tiempo producirían corrupción de memoria. El GIL es la solución simple a ese problema.

SIN GIL (situación de carrera en el contador de referencias)

Hilo 1:   lee ref_count = 1
Hilo 2:   lee ref_count = 1
Hilo 1:   escribe ref_count = 0  → libera el objeto
Hilo 2:   escribe ref_count = 0  → intenta liberar un objeto ya liberado
                                     → CRASH o corrupción silenciosa

CON GIL (solo un hilo modifica el contador a la vez)

Hilo 1:   ADQUIERE GIL
Hilo 1:   lee ref_count = 1, escribe ref_count = 0, libera objeto
Hilo 1:   LIBERA GIL
Hilo 2:   ADQUIERE GIL
Hilo 2:   lee ref_count = 0  → objeto ya no existe, no hace nada
Hilo 2:   LIBERA GIL
INFO
El GIL es una decisión de implementación de CPython, no del lenguaje Python. Otras implementaciones como Jython (Python en JVM) o PyPy-STM no tienen GIL. Pero CPython es el intérprete que usa el 99% del mundo, así que en la práctica el GIL existe.

Qué bloquea exactamente el GIL

El GIL bloquea la ejecución de bytecode Python. Cualquier instrucción que el intérprete de Python ejecute directamente está serializada. Esto incluye:

  • Operaciones aritméticas sobre enteros, floats, strings Python puros
  • Acceso a listas, diccionarios, sets
  • Llamadas a funciones Python
  • Creación y destrucción de objetos Python

Qué NO bloquea el GIL — y por qué threading sí funciona para I/O

El GIL se libera automáticamente durante operaciones que no ejecutan bytecode Python. Las más importantes:

  • Llamadas al sistema operativo (syscalls): cuando un hilo hace socket.recv(), file.read() o cualquier operación bloqueante, el intérprete libera el GIL antes de hacer la llamada al kernel. El hilo queda suspendido esperando la respuesta del SO, y otro hilo puede adquirir el GIL y ejecutar.
  • Extensiones C que lo liberan explícitamente: NumPy libera el GIL durante operaciones vectoriales. OpenSSL lo libera durante cifrado. La mayoría de librerías C bien escritas lo hacen.
  • time.sleep(): libera el GIL durante el sleep.
GIL durante I/O (por qué threading ayuda en IO-bound)

Tiempo →   0ms      10ms     20ms     30ms     40ms     50ms
           |--------|--------|--------|--------|--------|--------|

Hilo 1:    [PY ][  espera socket (GIL liberado)  ][PY ]
Hilo 2:          [PY ][  espera socket (GIL lib)  ][PY ]
Hilo 3:                [PY ][  espera socket  ][PY ]

GIL:       [H1 ][               rotando entre hilos              ]

↑ Mientras el hilo espera la respuesta de red, el GIL está libre
  y otros hilos ejecutan su código Python.
      
GIL durante CPU-bound (por qué threading NO ayuda)

Tiempo →   0ms      10ms     20ms     30ms     40ms     50ms
           |--------|--------|--------|--------|--------|--------|

Hilo 1:    [===PY bytecode===][===PY bytecode===][===PY ]
Hilo 2:                      [PY][              esperando GIL              ]
Hilo 3:                                          [PY][  esperando GIL      ]

GIL:       [====== Hilo 1 lo tiene casi siempre ======][ H2 ][ H3 ]

↑ El hilo activo nunca libera el GIL porque no hace I/O.
  Los demás hilos esperan, compiten, y el resultado es el mismo
  tiempo total + overhead de context switch.

Demostración observable: threading en CPU-bound no escala

Este script demuestra el efecto del GIL en CPU-bound. Ejecuta la misma función con 1 hilo, 2 hilos y 4 hilos. El tiempo total no mejora con más hilos — de hecho empeora ligeramente por el overhead del context switching.

python — efecto del GIL en CPU-bound con threading 
import threading
import time

def cuenta_regresiva(n):
    """Trabajo puro de CPU: decrementa un contador hasta cero."""
    while n > 0:
        n -= 1

N = 50_000_000  # 50 millones de decrementos

# ── 1 hilo: trabajo total = N
t0 = time.perf_counter()
t = threading.Thread(target=cuenta_regresiva, args=(N,))
t.start(); t.join()
t1_solo = time.perf_counter() - t0
print(f"1 hilo  (N=50M):    {t1_solo:.2f}s")

# ── 2 hilos: trabajo total = N (N/2 cada uno)
t0 = time.perf_counter()
hilos = [threading.Thread(target=cuenta_regresiva, args=(N // 2,)) for _ in range(2)]
for h in hilos: h.start()
for h in hilos: h.join()
t2_hilos = time.perf_counter() - t0
print(f"2 hilos (N/2 cada): {t2_hilos:.2f}s  (esperado: ~mitad, real: igual o más)")

# ── 4 hilos: trabajo total = N (N/4 cada uno)
t0 = time.perf_counter()
hilos = [threading.Thread(target=cuenta_regresiva, args=(N // 4,)) for _ in range(4)]
for h in hilos: h.start()
for h in hilos: h.join()
t4_hilos = time.perf_counter() - t0
print(f"4 hilos (N/4 cada): {t4_hilos:.2f}s  (esperado: ~cuarto, real: igual o más)")

print(f"\nEficiencia con 2 hilos: {t1_solo / t2_hilos:.2f}x  (ideal: 2.0x)")
print(f"Eficiencia con 4 hilos: {t1_solo / t4_hilos:.2f}x  (ideal: 4.0x)")

2 hilos (N/2 cada): 2.45s  (esperado: ~mitad, real: igual o más)

FAIL
Fallo observable: Añadir hilos a trabajo CPU-bound en Python no solo no mejora, sino que empeora el tiempo total. El overhead de context switching entre hilos (el SO sigue cambiando quién ejecuta) añade latencia sin ningún beneficio. Con 4 hilos el trabajo es un 15% más lento que con 1.

Cómo multiprocessing evita el GIL

Cada proceso Python tiene su propio intérprete y su propio GIL. Dos procesos Python no comparten GIL. Cuando lanzas 4 procesos con multiprocessing.Pool(4), cada proceso ejecuta de forma completamente independiente en su núcleo de CPU.

MULTIPROCESSING: cada proceso tiene su propio GIL

Núcleo 0:  [Proceso 1: GIL propio, ejecutando CPU-bound sin interferencia]
Núcleo 1:  [Proceso 2: GIL propio, ejecutando CPU-bound sin interferencia]
Núcleo 2:  [Proceso 3: GIL propio, ejecutando CPU-bound sin interferencia]
Núcleo 3:  [Proceso 4: GIL propio, ejecutando CPU-bound sin interferencia]

↑ Paralelismo real. Los 4 procesos ejecutan simultáneamente.
  No comparten memoria (cada uno tiene su heap propio).
python — multiprocessing sí escala en CPU-bound 
import multiprocessing
import time

def cuenta_regresiva(n):
    while n > 0:
        n -= 1

N = 50_000_000

# 1 proceso
t0 = time.perf_counter()
p = multiprocessing.Process(target=cuenta_regresiva, args=(N,))
p.start(); p.join()
print(f"1 proceso  (N=50M):    {time.perf_counter() - t0:.2f}s")

# 4 procesos, cada uno con N/4
t0 = time.perf_counter()
with multiprocessing.Pool(4) as pool:
    pool.map(cuenta_regresiva, [N // 4] * 4)
print(f"4 procesos (N/4 cada): {time.perf_counter() - t0:.2f}s  ← debería ser ~4x más rápido")

Output esperado (en máquina con 4+ núcleos):

output — multiprocessing sí escala 
1 proceso  (N=50M):    2.31s
4 procesos (N/4 cada): 0.63s  ← debería ser ~4x más rápido

El GIL y NumPy: por qué las operaciones vectoriales sí son paralelas

NumPy libera el GIL durante las operaciones vectoriales. Esto significa que si tienes código como resultado = np.dot(A, B), el GIL se libera mientras NumPy ejecuta la multiplicación de matrices en C, y otro hilo Python puede ejecutar en paralelo.

python — NumPy libera el GIL: threading sí ayuda con matrices 
import threading
import time
import numpy as np

def multiplicar_matrices():
    """NumPy libera el GIL: esta operación sí puede ser paralela con threading."""
    A = np.random.rand(2000, 2000)
    B = np.random.rand(2000, 2000)
    return np.dot(A, B)

# 1 hilo
t0 = time.perf_counter()
multiplicar_matrices()
print(f"1 hilo:   {time.perf_counter() - t0:.2f}s")

# 2 hilos simultáneos
t0 = time.perf_counter()
hilos = [threading.Thread(target=multiplicar_matrices) for _ in range(2)]
for h in hilos: h.start()
for h in hilos: h.join()
print(f"2 hilos:  {time.perf_counter() - t0:.2f}s  ← puede ser más rápido que 1 hilo")
WARN
Trampa con NumPy: NumPy ya usa internamente múltiples threads (OpenBLAS o MKL) para sus operaciones. Usar threading por encima puede no mejorar nada si NumPy ya satura los núcleos, o puede incluso empeorar por contención. Comprueba con np.show_config() cuántos threads usa tu instalación.

El futuro: el GIL en Python 3.13+

Python 3.13 introduce (como opción experimental) la posibilidad de compilar CPython sin GIL. Se activa con python3.13t (la variante "free-threaded"). En esta modalidad, el threading de Python sí ejecuta código Python en paralelo real.

Sin embargo, en 2026 esta característica todavía no es estable para producción. La mayor parte de extensiones C populares (NumPy, pandas) todavía requieren ajustes para funcionar correctamente sin GIL. El camino estándar para producción sigue siendo multiprocessing.

bash — verificar si tu Python es free-threaded (experimental) 
# Verificar versión y si es free-threaded
python3 --version
python3 -c "import sys; print('free-threaded:', sys._is_gil_enabled() == False)"

# Si usas conda, instalar la variante free-threaded (experimental)
conda install python=3.13 python-freethreading

Resumen: lo que el GIL significa para tus decisiones

Situación ¿El GIL importa? Herramienta correcta
Cálculo puro en Python (loops, dicts, listas) SÍ — bloquea el paralelismo multiprocessing
Peticiones HTTP, lectura de ficheros, sockets NO — el GIL se libera en I/O threading o asyncio
Operaciones NumPy/pandas vectoriales PARCIALMENTE — NumPy lo libera threading (con precaución) o multiprocessing
Extensiones C bien escritas (OpenSSL, zlib) NO — liberan el GIL explícitamente threading
Stress testing HTTP (Locust) NO — Locust usa gevent que parchea el I/O Locust directamente
