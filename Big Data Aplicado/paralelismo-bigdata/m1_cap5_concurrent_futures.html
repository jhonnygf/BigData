concurrent.futures: la API moderna

Una interfaz unificada para threading y multiprocessing. submit(), map(), as_completed() y cancelación de futures. La forma idiomática de paralelismo en Python moderno.

Por qué existe concurrent.futures

El módulo threading y multiprocessing dan control total pero requieren gestionar manualmente hilos/procesos, queues, joins y manejo de errores. concurrent.futures (añadido en Python 3.2) proporciona una interfaz de alto nivel que:

  • Unifica la API para hilos y procesos (solo cambia el executor)
  • Devuelve Future objects que representan resultados pendientes
  • Propaga excepciones automáticamente desde los workers al código principal
  • Gestiona el ciclo de vida del pool de forma automática con context manager
EXECUTOR

ThreadPoolExecutor

Pool de hilos. Para IO-bound. El GIL se libera en I/O. Hasta ~500 workers.

EXECUTOR

ProcessPoolExecutor

Pool de procesos. Para CPU-bound. Sin GIL compartido. Workers = núcleos físicos.

FUTURE

Future

Objeto que representa un resultado pendiente. Puede estar pending, running, done o cancelled.

submit(): enviar tareas individuales

executor.submit(funcion, *args, **kwargs) envía una tarea al pool y devuelve un Future inmediatamente. El pool la ejecuta en background. El código principal puede continuar o esperar con future.result().

python — submit básico con ThreadPoolExecutor 
from concurrent.futures import ThreadPoolExecutor
import time
import urllib.request

def descargar(url):
    """IO-bound: descarga una URL y devuelve el tamaño."""
    with urllib.request.urlopen(url, timeout=10) as r:
        return len(r.read())

URLS = [
    "https://httpbin.org/bytes/1024",
    "https://httpbin.org/bytes/2048",
    "https://httpbin.org/bytes/512",
    "https://httpbin.org/bytes/4096",
]

with ThreadPoolExecutor(max_workers=4) as executor:
    # submit devuelve Future inmediatamente
    futures = {executor.submit(descargar, url): url for url in URLS}

    for future in futures:
        url = futures[future]
        try:
            bytes_descargados = future.result()  # bloquea hasta que esté listo
            print(f"{url}: {bytes_descargados} bytes")
        except Exception as e:
            print(f"{url}: ERROR — {e}")

as_completed(): procesar resultados a medida que llegan

as_completed(futures) es un generador que yield los futures en el orden en que van terminando, no en el orden en que fueron enviados. Es el patrón correcto cuando quieres procesar resultados sin esperar a los más lentos.

python — as_completed: recoger resultados según terminan 
from concurrent.futures import ThreadPoolExecutor, as_completed
import time
import random

def procesar_endpoint(endpoint_id, latencia_simulada):
    """Simula una petición con latencia variable."""
    time.sleep(latencia_simulada)
    return {"id": endpoint_id, "latencia_ms": latencia_simulada * 1000}

tareas = [(i, random.uniform(0.1, 2.0)) for i in range(10)]

t0 = time.perf_counter()
with ThreadPoolExecutor(max_workers=10) as executor:
    futures = {
        executor.submit(procesar_endpoint, eid, lat): eid
        for eid, lat in tareas
    }

    # Los resultados llegan en orden de completado, no de envío
    for future in as_completed(futures):
        endpoint_id = futures[future]
        resultado = future.result()
        elapsed = time.perf_counter() - t0
        print(f"t+{elapsed:.2f}s: endpoint {endpoint_id} → {resultado['latencia_ms']:.0f}ms")

print(f"\nTotal: {time.perf_counter()-t0:.2f}s (secuencial hubiera tardado ~{sum(l for _,l in tareas):.1f}s)")

Output (los más rápidos aparecen primero):

output
t+0.12s: endpoint 3 → 112ms
t+0.34s: endpoint 7 → 338ms
t+0.51s: endpoint 1 → 507ms
...
t+1.93s: endpoint 9 → 1928ms

Total: 1.95s (secuencial hubiera tardado ~9.7s)

executor.map(): la forma más simple

executor.map(funcion, iterable) es similar a pool.map(): aplica la función a cada elemento, devuelve una lista de resultados en el mismo orden de entrada. A diferencia de as_completed, bloquea y preserva el orden.

python — executor.map con timeout global 
from concurrent.futures import ThreadPoolExecutor
import urllib.request

def fetch(url):
    with urllib.request.urlopen(url, timeout=5) as r:
        return r.status

urls = ["https://httpbin.org/status/200"] * 8

with ThreadPoolExecutor(max_workers=8) as executor:
    # timeout=10: si algún future no termina en 10s, lanza TimeoutError
    resultados = list(executor.map(fetch, urls, timeout=10))

print(f"Respuestas: {resultados}")

ProcessPoolExecutor: el mismo API, paralelismo real

Cambiar de ThreadPoolExecutor a ProcessPoolExecutor es una línea de código. El API es idéntico. La diferencia interna: los workers son procesos, no hilos.

python — ProcessPoolExecutor para CPU-bound 
from concurrent.futures import ProcessPoolExecutor, as_completed
import hashlib
import time

def hashear_bloque(bloque_id, n_hashes):
    """CPU-bound: genera n_hashes SHA-256 consecutivos."""
    h = str(bloque_id).encode()
    for _ in range(n_hashes):
        h = hashlib.sha256(h).digest()
    return bloque_id, h.hex()

BLOQUES = list(range(16))      # 16 bloques de trabajo
N_HASHES = 200_000             # hashes por bloque

if __name__ == "__main__":
    t0 = time.perf_counter()
    resultados = {}

    with ProcessPoolExecutor() as executor:
        futures = {executor.submit(hashear_bloque, b, N_HASHES): b for b in BLOQUES}
        for future in as_completed(futures):
            bloque_id, hash_final = future.result()
            resultados[bloque_id] = hash_final

    print(f"{len(BLOQUES)} bloques procesados en {time.perf_counter()-t0:.2f}s")
    print(f"Muestra: bloque 0 → {resultados[0][:32]}...")

Manejo de errores: excepciones en futures

Cuando un worker lanza una excepción, el Future la captura y la re-lanza cuando el código principal llama a .result(). Esto hace el manejo de errores mucho más limpio que con callbacks.

python — capturar errores por future sin romper los demás 
from concurrent.futures import ThreadPoolExecutor, as_completed
import urllib.request

def fetch_seguro(url):
    with urllib.request.urlopen(url, timeout=5) as r:
        return r.status

urls_mixtas = [
    "https://httpbin.org/status/200",
    "https://httpbin.org/status/404",  # lanzará HTTPError
    "https://host-que-no-existe.xyz",  # lanzará URLError
    "https://httpbin.org/status/500",  # lanzará HTTPError
    "https://httpbin.org/status/200",
]

ok = 0
errores = []

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(fetch_seguro, url): url for url in urls_mixtas}

    for future in as_completed(futures):
        url = futures[future]
        try:
            status = future.result()
            ok += 1
            print(f"OK    {url} → {status}")
        except Exception as e:
            errores.append({"url": url, "error": str(e)})
            print(f"ERROR {url} → {e}")

print(f"\nResumen: {ok} OK, {len(errores)} errores")
INFO
Ventaja sobre threading raw: Con threading.Thread, si el worker lanza una excepción y nadie la captura, se imprime en stderr pero el hilo termina silenciosamente. Con Future.result(), la excepción se propaga de forma explícita al hilo principal y no puede ser ignorada accidentalmente.

Cancelar futures

Un future puede cancelarse si todavía no ha empezado a ejecutarse. Una vez que un worker lo ha recogido, no puede cancelarse (el worker ya está ejecutando).

python — cancelar futures pendientes si uno falla 
from concurrent.futures import ThreadPoolExecutor
import time

def tarea_larga(id, duracion):
    print(f"Tarea {id}: arrancando ({duracion}s)")
    time.sleep(duracion)
    if id == 3:
        raise ValueError(f"Tarea {id} falló deliberadamente")
    return f"tarea_{id}_ok"

with ThreadPoolExecutor(max_workers=2) as executor:
    futures = [executor.submit(tarea_larga, i, 1 + i * 0.5) for i in range(8)]

    for future in futures:
        try:
            resultado = future.result(timeout=5)
            print(f"Resultado: {resultado}")
        except ValueError as e:
            print(f"Fallo crítico: {e}")
            # Cancelar todas las tareas pendientes
            canceladas = sum(1 for f in futures if f.cancel())
            print(f"Canceladas {canceladas} tareas pendientes")
            break

wait(): esperar un subconjunto de futures

python — wait con FIRST_COMPLETED para reacción rápida 
from concurrent.futures import ThreadPoolExecutor, wait, FIRST_COMPLETED
import time
import random

def tarea(n):
    tiempo = random.uniform(0.5, 3.0)
    time.sleep(tiempo)
    return n, tiempo

with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(tarea, i) for i in range(5)}

    while futures:
        # Espera hasta que al menos uno termine
        done, futures = wait(futures, return_when=FIRST_COMPLETED)
        for f in done:
            n, t = f.result()
            print(f"Tarea {n} terminó en {t:.2f}s. Quedan {len(futures)} pendientes.")

Elegir el número correcto de workers

ExecutorRegla de baseAjuste
ThreadPoolExecutor min(32, os.cpu_count() + 4) (default Python 3.8+) Para IO-bound: subir hasta 100-500 según latencia del servicio remoto
ProcessPoolExecutor os.cpu_count() (default) Nunca superar el número de núcleos físicos; más procesos = más overhead sin ganancia
python — encontrar el número óptimo de workers para IO-bound 
from concurrent.futures import ThreadPoolExecutor
import time
import urllib.request

def ping(url):
    with urllib.request.urlopen(url, timeout=5) as r:
        return r.status

URL = "https://httpbin.org/status/200"
N_REQUESTS = 50

for n_workers in [1, 5, 10, 20, 50]:
    t0 = time.perf_counter()
    with ThreadPoolExecutor(max_workers=n_workers) as ex:
        list(ex.map(ping, [URL] * N_REQUESTS))
    elapsed = time.perf_counter() - t0
    rps = N_REQUESTS / elapsed
    print(f"Workers={n_workers:3d}: {elapsed:.2f}s → {rps:.1f} req/s")

Output típico (la ganancia se satura en algún punto):

output
Workers=  1: 52.3s → 0.96 req/s
Workers=  5: 11.1s → 4.5 req/s
Workers= 10:  5.8s → 8.6 req/s
Workers= 20:  3.2s → 15.6 req/s
Workers= 50:  2.1s → 23.8 req/s  ← el límite lo pone el servidor, no el cliente
WARN
Más workers no siempre significa más velocidad. El límite puede estar en el servidor destino (rate limiting, conexiones máximas), en el ancho de banda de red, o en el overhead de crear y gestionar demasiados hilos. Mide siempre en el entorno real.
