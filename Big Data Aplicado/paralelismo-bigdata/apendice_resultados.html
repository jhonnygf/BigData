1 · Percentiles — qué significan realmente

La aritmética detrás del percentil

Un percentil pN dice: "el N% de las peticiones tardaron menos o igual que este valor". Para calcularlo: ordena todas las latencias de menor a mayor, toma el valor en la posición N/100 × total de la lista.

Cálculo manual — Python 
latencias = sorted([45, 12, 89, 34, 67, 203, 55, 78, 91, 44,
                    38, 502, 29, 61, 73, 82, 99, 141, 56, 48])

n = len(latencias)   # 20 peticiones

p50 = latencias[int(n * 0.50)]  # posición 10 → 61 ms
p95 = latencias[int(n * 0.95)]  # posición 19 → 203 ms (casi el peor)
p99 = latencias[int(n * 0.99)]  # posición 19 → 203 ms (con N=20, redondea igual)

print(f"p50={p50}ms  p95={p95}ms  p99={p99}ms  max={latencias[-1]}ms")
# p50=61ms  p95=203ms  p99=203ms  max=502ms

# Con N=1000+, p99 y max divergen significativamente:
# p50=58ms  p95=180ms  p99=420ms  max=3400ms  ← la cola larga es real

Lo que no te dice la media

Si p50 = 50ms pero p99 = 2000ms, la media puede ser 80ms. Esa media es inútil para dimensionar sistemas. El 1% de usuarios que experimenta 2000ms es el que abre un ticket de soporte. Los SLAs de Big Data siempre se definen en percentiles:

PercentilQué describeSLA habitual Big Data
p50 (mediana) Experiencia del usuario mediano < 100ms para APIs REST, < 10ms para Kafka
p95 Experiencia del 95% El umbral habitual de SLA. < 300ms REST, < 50ms Kafka
p99 Peor caso excluido el 1% < 1s REST, < 200ms Kafka. Si supera esto, investigar.
p999 (0.1%) Outliers extremos Raramente en SLA, útil para detección de GC pauses en JVM
max Peor petición absoluta Diagnóstico, no SLA. Puede ser timeout de un reconexión.

La trampa del warm-up

WARN Las primeras peticiones de cualquier test son anómalas: JVM compilando (JIT), pools de conexión abriéndose, caches frías. Descartar el primer 5-10% de peticiones de las métricas. Locust y Gatling permiten configurar un período de warm-up. En el StressTester propio del Cap 9 y el JucStressTester del Cap 11 hay un contador de warm-up explícito.

2 · Identificar el cuello de botella

El cuello de botella puede estar en cuatro lugares distintos. La clave es aislarlos sistemáticamente. Un test de stress que sube latencias sin identificar dónde no sirve para tomar decisiones de ingeniería.

Red / Conectividad

  • Latencia base alta (p50 > 200ms en LAN)
  • Timeouts de conexión TCP
  • Throughput estabilizado muy por debajo del esperado
  • Packet loss en el generador de carga

Generador de carga

  • CPU del generador al 100%
  • Throughput no sube aunque el servidor aguante
  • Error rate artificial (no del servicio real)
  • Memoria del generador saturada

Aplicación / API

  • p99 sube con la carga mientras p50 aguanta
  • 5xx en errores (no timeouts)
  • CPU de la app al 100%
  • Thread pool exhausto (cola de peticiones esperando)

Base de datos / Kafka

  • Latencias de app normales pero Kafka lag crece
  • p99 sube sólo en escrituras, no en lecturas
  • MongoDB / HDFS IO wait alto
  • GC pauses en brokers (picos en p999)

Protocolo de diagnóstico — qué monitorear mientras corre el test

Comandos a ejecutar durante el test (terminales separadas) 
# ── Terminal 1: CPU y memoria del servidor ────────────────────
htop   # o top -d 1

# ── Terminal 2: CPU del generador de carga ────────────────────
top -d 1 -p $(pgrep -f locust)    # Locust
top -d 1 -p $(pgrep -f gatling)   # Gatling
# Si el generador usa > 80% CPU, los resultados son inválidos

# ── Terminal 3: red ────────────────────────────────────────────
iftop -i eth0          # tráfico en tiempo real
netstat -s | grep -E "retransmit|failed"  # pérdida de paquetes
ss -s                  # estado de sockets TCP

# ── Terminal 4: Kafka lag ─────────────────────────────────────
# Kafka CLI
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group stress-consumer-group | column -t

# ── Terminal 5: JVM GC (si la app es Java) ───────────────────
jstat -gcutil $(pgrep -f tu-app.jar) 1000  # cada segundo
# FGCT (Full GC time) creciendo = GC pressure

# ── Terminal 6: MongoDB (si aplica) ──────────────────────────
mongotop 1             # colecciones más activas
mongostat 1            # inserts/queries/updates por segundo

Árbol de decisión — dónde está el problema

  ¿Throughput del generador llega al objetivo?
  │
  ├── NO → ¿CPU generador > 80%?
  │         ├── SÍ → Generador es el cuello. Usar Gatling o modo distribuido.
  │         └── NO → ¿Errores de conexión TCP?
  │                   ├── SÍ → Problema de red o límite de sockets del OS
  │                   └── NO → Investigar logs del generador
  │
  └── SÍ → ¿p95 o p99 sube con la carga?
            │
            ├── NO → Sistema aguanta. Subir carga hasta encontrar el límite.
            │
            └── SÍ → ¿CPU servidor > 80%?
                      ├── SÍ → Cuello en la aplicación (CPU-bound)
                      │         Acciones: escalar horizontalmente, optimizar código
                      └── NO → ¿Kafka lag creciendo?
                                ├── SÍ → Cuello en Kafka/consumers
                                │         Acciones: más particiones, más consumers
                                └── NO → ¿IO wait > 30% en servidor?
                                          ├── SÍ → Cuello en disco/DB
                                          └── NO → Thread starvation (revisar pool sizes)

El límite del generador — cómo detectarlo

FAIL El error más frecuente en stress testing: concluir que "el sistema aguanta X req/s" cuando en realidad era el generador el que se saturó antes. Síntoma: p95 del servidor se ve bien pero el throughput no sube aunque aumente los usuarios. Verificación: mira CPU del proceso generador. Si supera 80%, tus resultados son inválidos.
Verificar si el generador es el cuello — Python 
import psutil, os

proc = psutil.Process(os.getpid())

# Durante el test, en un hilo separado:
def monitor_generador():
    while True:
        cpu  = proc.cpu_percent(interval=1)
        mem  = proc.memory_info().rss / 1024 / 1024  # MB
        conn = len(proc.connections())
        print(f"[GEN] CPU={cpu:.1f}%  MEM={mem:.0f}MB  SOCKETS={conn}")
        if cpu > 80:
            print("WARN: generador al límite — resultados pueden estar afectados")
        time.sleep(5)

3 · Patrones de degradación — cómo se ve cada problema

Patrón 1 — Escalón: el sistema se rompe de golpe

  Latencia (ms)
  500 │                                              ╭────────────── p99
  400 │                                        ╭─────╯
  300 │                                  ╭─────╯               ← punto de quiebre
  200 │                                  │
  100 │ ───────────────────────────────────                    p95
   50 │ ─────────────────────────────────────────────────────  p50
      └─────────────────────────────────────────────────────────► usuarios
           50        100       150       200       250

  Causa típica: pool de threads/conexiones tiene un límite fijo.
  Hasta 150 usuarios → todo ok (pool tiene capacidad).
  A partir de 150 → cola de espera crece → latencias explotan.
  Acción: aumentar pool size o escalar instancias.

Patrón 2 — Degradación gradual: memory leak o acumulación de recursos

  Latencia (ms) — carga constante en el tiempo
  400 │                                              ╭─── p99
  300 │                                        ╭─────╯
  200 │                                  ╭─────╯
  100 │                            ╭─────╯
   80 │ ──────────────────────────────────────────── p95
   50 │ ──────────────────────────────────────────── p50
      └─────────────────────────────────────────────────────────► tiempo
        5min     10min    15min    20min    25min

  Causa típica: memoria creciendo (leak), GC full GC cada vez más frecuente,
  conexiones no cerradas acumulándose.
  Acción: heap dump, revisar cierres de conexión, perfilar memoria.

Patrón 3 — Picos periódicos: GC pauses o cron jobs

  Latencia (ms)
  2000│                      ↑                      ↑
  1500│                      ║                      ║    ← picos p99/p999
   500│         ↑            ║         ↑            ║
   300│         ║            ║         ║            ║
   100│──────────────────────────────────────────────── p95
    50│──────────────────────────────────────────────── p50
      └─────────────────────────────────────────────────────────► tiempo
        0s      30s      60s      90s     120s

  Causa típica: Full GC de JVM (Java app) o cron job en el servidor.
  En JVM: jstat -gcutil muestra picos en FGCT.
  Acción: tuning de JVM heap, cambiar GC a G1GC o ZGC.

4 · Checklist antes de estresar un clúster real

WARN Un test de stress mal ejecutado puede: llenar discos de logs, saturar colas de Kafka y afectar consumidores de producción, agotar connection pools de bases de datos, o provocar reequilibrios de liderazgo en Kafka que afecten al tráfico real. Revisa esta lista antes de disparar cualquier test contra sistemas compartidos.

Entorno y aislamiento

Confirmar que el test apunta a un entorno de staging/testing, no producción
Verificar la URL base en el script. Doble-check con ping/curl antes de escalar.
Topics de Kafka exclusivos para el test
Crear topics con prefijo "stress-" separados de los de producción.
Consumer groups de test separados
Un consumer group de test que consume el topic de stress, no el de producción.
Quotas de Kafka configuradas para el producer de test
producer_byte_rate limitado para no monopolizar el broker compartido.

Capacidad del generador de carga

El generador tiene suficientes recursos para el throughput objetivo
Regla general: 1 core de generador → ~10.000 req/s en Gatling, ~1.000 req/s en Locust básico.
Límite de file descriptors (ulimit) aumentado
ulimit -n 65535 — por defecto 1024 es insuficiente para miles de conexiones.
Verificar que el generador y el servidor están en la misma red (latencia de red mínima)
Si el generador está en internet y el servidor en LAN, estás midiendo internet, no el sistema.

Monitoreo durante el test

Métricas del servidor abiertas antes de arrancar el test
htop, iftop, y si hay Grafana/Prometheus: abrir el dashboard antes del disparo.
Kafka consumer lag monitoring activo
kafka-consumer-groups.sh --describe cada 10s o con Kafka Manager/Cruise Control.
CPU del generador de carga monitorizado durante el test
Si supera 80%, los resultados son inválidos. Abortar y escalar el generador.
Plan de aborto: saber cómo parar el test inmediatamente
Locust: Ctrl+C. Gatling: Ctrl+C. Tener el comando listo antes de empezar.

Después del test

Limpiar datos de prueba generados
Borrar mensajes de topics de test, documentos de MongoDB de test, etc.
Verificar que el sistema volvió a su estado normal
Kafka lag a 0, latencias de producción dentro de SLA, no hay alarmas activas.
Documentar resultados: throughput máximo sostenido, punto de quiebre, p95 y p99 en ese punto
Sin documentación, el test siguiente no tiene baseline para comparar.

5 · Referencia rápida de comandos de diagnóstico

Sistema operativo 
# CPU por proceso
top -d 1 -p PID
pidstat -u 1  # por proceso, cada segundo

# Memoria
free -h
cat /proc/meminfo | grep -E "MemFree|Buffers|Cached"

# Conexiones TCP
ss -s                          # resumen
ss -tn state ESTABLISHED       # conexiones activas
ss -tn | wc -l                 # total
netstat -i                     # errores por interfaz

# File descriptors
lsof -p PID | wc -l            # FDs del proceso
cat /proc/sys/fs/file-nr       # sistema total

# Disco / IO
iostat -x 1                    # IO por dispositivo, cada segundo
iotop                          # IO por proceso

# Ajustar límites para tests de alta concurrencia
ulimit -n 65535                # file descriptors (sesión actual)
echo "* soft nofile 65535" >> /etc/security/limits.conf
sysctl -w net.core.somaxconn=65535  # backlog de conexiones TCP
Kafka 
# Lag de consumer group
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group mi-consumer-group

# Throughput de un topic (mensajes/s)
kafka-consumer-perf-test.sh \
  --bootstrap-server localhost:9092 \
  --topic stress-test \
  --messages 100000 \
  --threads 4

# Throughput del producer
kafka-producer-perf-test.sh \
  --topic stress-test \
  --num-records 100000 \
  --record-size 1000 \
  --throughput 10000 \
  --producer-props bootstrap.servers=localhost:9092

# Estado de los brokers
kafka-broker-api-versions.sh --bootstrap-server localhost:9092

# Latencia de red al broker
kafkacat -b localhost:9092 -L  # listar metadatos (instala kafkacat/kcat)
JVM (Java apps) 
# GC en tiempo real
jstat -gcutil PID 1000          # cada segundo
# FGCT creciendo = Full GC frecuente = problema de memoria

# Heap dump para análisis posterior
jmap -dump:format=b,file=heap.hprof PID
# Analizar con VisualVM o Eclipse MAT

# Thread dump (detectar deadlocks o thread starvation)
jstack PID > threads.txt
# Buscar "BLOCKED", "WAITING", "deadlock" en el fichero

# Monitoreo continuo con JVM Flags recomendados para stress
java -server \
     -XX:+UseG1GC \
     -Xms4g -Xmx4g \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -Xloggc:/tmp/gc.log \
     -jar tu-app.jar

6 · Glosario de términos

TérminoDefinición
Throughput Peticiones completadas por segundo. La métrica de capacidad.
Latencia Tiempo desde que se envía la petición hasta que se recibe la respuesta completa.
Percentil (pN) N% de las peticiones tuvieron latencia ≤ este valor.
Saturación Punto donde añadir más carga ya no aumenta el throughput — el sistema está al límite.
Backpressure Mecanismo por el que un sistema saturado señaliza al productor que reduzca la tasa.
SLA (Service Level Agreement) Compromiso formal de rendimiento: "p95 < 300ms con 99.9% de disponibilidad".
Warm-up Período inicial donde la JVM compila, los pools se abren y las caches se llenan. No representa el estado estable.
Consumer lag En Kafka: diferencia entre el último offset producido y el último leído por el consumer. Si crece, el consumer no da abasto.
GC pause Parada del proceso Java para recoger basura. Full GC puede durar 100ms-2s y aparece como pico en p999.
Thread starvation El pool de threads está lleno; nuevas peticiones esperan en cola. Se ve como latencias que suben aunque CPU esté baja.
Open model Usuarios llegan a tasa fija independientemente de si los anteriores terminaron. Modela tráfico web real.
Closed model Número constante de usuarios activos. Modela thread pools, sistemas con backpressure.
