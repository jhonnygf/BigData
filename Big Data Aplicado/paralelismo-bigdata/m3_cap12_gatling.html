Gatling

DSL declarativo · scenarios · feeders · throttle · informes HTML

Por qué Gatling para Big Data

Locust es perfecto cuando ya sabes Python y necesitas flexibilidad. Pero cuando necesitas estresar un clúster de producción con cientos de miles de peticiones por segundo desde una sola máquina, Gatling es otra categoría. Basado en Akka + Netty, usa actores y canales NIO asíncronos — no hilos por usuario. Un proceso Gatling en un servidor de 8 cores puede generar fácilmente 100.000 req/s contra un API REST o un clúster Kafka.

Motor

Akka Actors + Netty NIO. No hay un hilo por usuario virtual — actores ligeros gestionados por el scheduler de Akka.

DSL Scala

Las simulaciones se escriben en Scala. No necesitas saber Scala avanzado — el DSL se lee casi como inglés.

Informes HTML

Genera automáticamente un informe interactivo con gráficas de percentiles, distribución de respuestas y errores.

Integración CI

Plugin Maven/Gradle. Se ejecuta en Jenkins/GitLab CI con umbrales de fallo configurables.

INFO Gatling usa Scala pero desde la versión 3.7 ofrece también una API Java. En este capítulo usamos Scala para las simulaciones (es el estándar de la comunidad) y Java/Maven para la integración CI. No hace falta dominar Scala — el DSL de Gatling es deliberadamente simple.

2 · El DSL de Gatling — capas

Una simulación Gatling se construye en cuatro capas que se encajan como bloques. Entender la jerarquía es clave para no mezclarlas.

Simulation
Clase que extiende Simulation. El punto de entrada. Define el protocolo HTTP, los escenarios y la inyección de carga.
ScenarioBuilder
Define el flujo de un usuario virtual: qué pasos ejecuta, en qué orden, con qué pausas. Se crea con scenario("nombre").
ChainBuilder
Secuencia de acciones: exec(), pause(), repeat(), doIf(), feed(). Se encadena con .exec().
HttpRequestBuilder
Una petición HTTP individual: http("nombre").get(url), con headers, body, checks y extracción de valores.

Simulación mínima funcional

MinimalSimulation.scala 
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class MinimalSimulation extends Simulation {

  // ── Protocolo HTTP base ──────────────────────────────────────
  val protocolo = http
    .baseUrl("http://localhost:8000")
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")

  // ── Escenario ────────────────────────────────────────────────
  val escenario = scenario("Hello")
    .exec(
      http("GET /hello")
        .get("/hello")
        .check(status.is(200))
    )
    .pause(1, 3)  // pausa aleatoria entre 1 y 3 segundos

  // ── Inyección de carga ───────────────────────────────────────
  setUp(
    escenario.inject(
      atOnceUsers(10)  // 10 usuarios simultáneos al inicio
    )
  ).protocols(protocolo)
}

3 · Scenarios — flujos complejos

Un escenario real de Big Data no es una sola petición en bucle. Hay flujos multi-paso: autenticarse, leer un batch, escribir un resultado, verificar el estado. Gatling soporta todo esto de forma declarativa.

Checks y extracción de valores

check() valida la respuesta y puede extraer valores para usarlos en pasos posteriores. Es el equivalente a catch_response en Locust.

FlujoBigData.scala — autenticación + consulta 
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class FlujoBigData extends Simulation {

  val protocolo = http
    .baseUrl("http://api-bigdata.local:8080")
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")

  // ── Paso 1: autenticación → extrae token ────────────────────
  val autenticar = exec(
    http("POST /auth/login")
      .post("/auth/login")
      .body(StringBody("""{"user":"stress","pass":"test123"}"""))
      .check(
        status.is(200),
        jsonPath("$.token").saveAs("authToken")  // extrae y guarda en sesión
      )
  ).exitHereIfFailed  // si login falla, el usuario no continúa

  // ── Paso 2: enviar evento a Kafka via REST ──────────────────
  val enviarEvento = exec(
    http("POST /kafka/produce")
      .post("/kafka/produce")
      .header("Authorization", "Bearer #{authToken}")  // usa el token extraído
      .body(StringBody(
        """{"topic":"stress-test","data":{"ts":"#{timestamp}","val":#{random}}}"""
      ))
      .check(
        status.is(202),
        jsonPath("$.offset").exists  // verifica que Kafka asignó offset
      )
  )

  // ── Paso 3: consultar el estado del consumer group ──────────
  val verificarConsumer = exec(
    http("GET /kafka/consumer-groups/stress-group")
      .get("/kafka/consumer-groups/stress-group")
      .header("Authorization", "Bearer #{authToken}")
      .check(
        status.is(200),
        jsonPath("$.lag").lte(1000)  // lag máximo aceptable: 1000 mensajes
      )
  )

  // ── Escenario completo ───────────────────────────────────────
  val escenario = scenario("Flujo Big Data")
    .exec(autenticar)
    .pause(500.milliseconds)
    .repeat(50) {          // cada usuario envía 50 eventos
      exec(
        session => session.set("timestamp", System.currentTimeMillis())
                          .set("random", (Math.random() * 1000).toInt)
      )
      .exec(enviarEvento)
      .pause(100.milliseconds, 500.milliseconds)
    }
    .exec(verificarConsumer)

  // ── Carga ────────────────────────────────────────────────────
  setUp(
    escenario.inject(
      rampUsers(100).during(30.seconds)  // 100 usuarios en 30 segundos
    )
  ).protocols(protocolo)
   .assertions(
     global.responseTime.percentile(95).lte(200),
     global.failedRequests.percent.lte(2)
   )
}
INFO #{variable} es la sintaxis de interpolación de sesión en Gatling (Gatling EL). Cada usuario virtual tiene su propia sesión (Session) donde se guardan los valores extraídos con saveAs. No es compartida entre usuarios — es thread-safe por diseño.

4 · Feeders — datos realistas

Un feeder es una fuente de datos que alimenta las peticiones con valores variables. Sin feeders, todos los usuarios envían los mismos datos — no es realista y puede activar caches que enmascaren la carga real.

Tipos de feeder

TipoSintaxisComportamiento
CSV circular csv("datos.csv").circular Vuelve al inicio cuando se agota. Más usado.
CSV aleatorio csv("datos.csv").random Selecciona fila aleatoria. Evita patrones secuenciales.
JSON jsonFile("datos.json").circular Igual que CSV pero con JSON. Útil para payloads complejos.
Programático Iterator.continually(Map(...)) Genera datos en código. Máxima flexibilidad.
recursos/usuarios.csv 
userId,topic,particion
u001,events-raw,0
u002,events-raw,1
u003,events-processed,0
u004,events-raw,2
u005,events-processed,1
FeederSimulation.scala 
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class FeederSimulation extends Simulation {

  // ── Feeder desde CSV ─────────────────────────────────────────
  val csvFeeder = csv("usuarios.csv").circular

  // ── Feeder programático — IDs únicos ─────────────────────────
  val idFeeder = Iterator.continually(
    Map("msgId" -> java.util.UUID.randomUUID().toString)
  )

  val protocolo = http.baseUrl("http://api-bigdata.local:8080")
    .contentTypeHeader("application/json")

  val escenario = scenario("Carga con Feeder")
    .feed(csvFeeder)   // inyecta userId, topic, particion en sesión
    .feed(idFeeder)    // inyecta msgId único
    .exec(
      http("Produce mensaje")
        .post("/kafka/produce")
        .body(StringBody(
          """{
            "msgId": "#{msgId}",
            "userId": "#{userId}",
            "topic": "#{topic}",
            "partition": #{particion}
          }"""
        ))
        .check(status.is(202))
    )

  setUp(
    escenario.inject(rampUsers(200).during(60.seconds))
  ).protocols(protocolo)
}
WARN Si el CSV tiene pocas filas y muchos usuarios, con .queue (por defecto) los últimos usuarios se quedarán sin datos y fallarán. Usa siempre .circular o .random para tests de larga duración.

5 · Perfiles de carga e inyección — throttle

El perfil de carga define cuántos usuarios se inyectan y cuándo. Es la diferencia entre un test de capacidad (¿cuánto aguanta?), un test de pico (¿qué pasa con un spike repentino?) y un test de resistencia (¿se degrada con el tiempo?).

Modelos de inyección — open vs closed

Perfiles de inyección 
// ── Modelo abierto (usuarios entran, pueden salir) ──────────
setUp(escenario.inject(
  nothingFor(5.seconds),                  // warm-up silencioso
  atOnceUsers(20),                        // burst inicial: 20 de golpe
  rampUsers(80).during(30.seconds),       // sube a 80 en 30 s
  constantUsersPerSec(10).during(60.seconds),  // 10 nuevos/s durante 1 min
  rampUsersPerSec(10).to(50).during(30.seconds),  // sube de 10 a 50/s
  stressPeakUsers(200).during(20.seconds) // pico masivo en 20 s
))

// ── Modelo cerrado (concurrencia constante) ──────────────────
// Útil para emular thread pools de tamaño fijo
setUp(escenario.inject(
  constantConcurrentUsers(50).during(2.minutes),  // 50 usuarios activos siempre
  rampConcurrentUsers(10).to(100).during(60.seconds)
))

// ── throttle — límite de req/s independiente de usuarios ─────
// Útil cuando el sistema tiene rate limits conocidos
setUp(escenario.inject(atOnceUsers(1000)))
  .throttle(
    reachRps(100).in(10.seconds),    // sube a 100 rps en 10 s
    holdFor(1.minute),               // mantiene 100 rps durante 1 min
    jumpToRps(200),                  // salta a 200 rps
    holdFor(30.seconds)              // mantiene 200 rps
  )
  .protocols(protocolo)
INFO Modelo abierto: los usuarios entran a una tasa fija independientemente de si los anteriores han terminado. Modela sistemas web reales donde los usuarios no esperan. Modelo cerrado: siempre hay exactamente N usuarios activos. Modela thread pools, connection pools o sistemas con backpressure. La elección afecta drásticamente el comportamiento observado bajo saturación.

Ejemplo — test de capacidad completo para clúster Kafka/REST

CapacidadBigData.scala 
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class CapacidadBigData extends Simulation {

  val baseUrl   = System.getProperty("baseUrl", "http://localhost:8080")
  val maxUsers  = System.getProperty("maxUsers", "500").toInt
  val duracion  = System.getProperty("duracion", "120").toInt.seconds

  val protocolo = http
    .baseUrl(baseUrl)
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")
    .shareConnections  // reutiliza conexiones TCP entre usuarios

  val feederEventos = Iterator.continually(Map(
    "eventId"   -> java.util.UUID.randomUUID().toString,
    "timestamp" -> System.currentTimeMillis(),
    "payload"   -> s"""{"sensor":"s${(Math.random()*100).toInt}","val":${(Math.random()*100).toInt}}"""
  ))

  val producirEvento = exec(
    http("POST /events")
      .post("/events")
      .body(StringBody(
        """{"id":"#{eventId}","ts":#{timestamp},"data":#{payload}}"""
      ))
      .check(
        status.in(200, 201, 202),
        responseTimeInMillis.lte(500)  // falla si la respuesta tarda > 500ms
      )
  )

  val consultarMetricas = exec(
    http("GET /metrics/kafka")
      .get("/metrics/kafka")
      .check(
        status.is(200),
        jsonPath("$.consumer_lag").ofType[Int].lte(5000)
      )
  )

  val escenarioProduccion = scenario("Producción continua")
    .feed(feederEventos)
    .exec(producirEvento)
    .pause(50.milliseconds, 200.milliseconds)

  val escenarioMonitor = scenario("Monitoring")
    .exec(consultarMetricas)
    .pause(5.seconds)

  setUp(
    // 90% productores, 10% monitoring
    escenarioProduccion.inject(
      nothingFor(5.seconds),
      rampUsers((maxUsers * 0.9).toInt).during(30.seconds),
      constantUsersPerSec(2).during(duracion)
    ),
    escenarioMonitor.inject(
      nothingFor(10.seconds),
      constantConcurrentUsers((maxUsers * 0.1).toInt).during(duracion)
    )
  ).protocols(protocolo)
   .assertions(
     global.responseTime.percentile(95).lte(300),
     global.responseTime.percentile(99).lte(500),
     global.failedRequests.percent.lte(1),
     forAll.responseTime.max.lte(2000)  // ninguna petición > 2s
   )
}
Ejecutar con parámetros 
# Standalone
./bin/gatling.sh -s CapacidadBigData \
  -DbaseUrl=http://api-prod:8080 \
  -DmaxUsers=1000 \
  -Dduracion=300

# Maven

6 · Leer el informe HTML de Gatling

El informe HTML es la principal razón por la que muchos equipos eligen Gatling sobre alternativas. Se genera automáticamente en results/ y no necesita servidor — ábrelo con el navegador directamente.

p50 Mediana real de usuarios
p95 Umbral SLA habitual
p99 Peor caso excluido 1%
req/s Throughput real alcanzado

Secciones del informe y qué buscar

Sección Qué muestra Señal de alarma
Global Statistics Totales: peticiones, OK/KO, percentiles globales KO% > 1%, p99 > SLA
Response Time Distribution Histograma de latencias Distribución bimodal: dos picos separados indican dos poblaciones (cacheadas vs no)
Response Time Percentiles over Time Evolución de p50/p95/p99 durante el test Latencias que suben progresivamente = cola creciendo, sistema saturado
Active Users over Time Usuarios activos por segundo Diferencia entre usuarios inyectados y activos = usuarios esperando (backpressure)
Requests/s over Time Throughput real vs esperado Throughput que se estabiliza por debajo del máximo = cuello de botella identificado
Errors Distribución de errores por tipo Timeouts de conexión = red o backlog; 5xx = aplicación; checks fallidos = lógica negocio

Interpretar la gráfica de percentiles en el tiempo

  Latencia (ms)
  400 │                                        ╭─── p99
  350 │                                  ╭─────╯
  300 │                           ╭──────╯           ← sistema saturado
  250 │                    ╭──────╯
  200 │                                        ╭─── p95
  150 │             ╭──────────────────────────╯
  100 │ ───────────────────────────────────────── p50
   50 │
      └─────────────────────────────────────────────► tiempo
           fase normal    saturación empieza aquí ↑

  Señal: p99 diverge de p50 = percentil alto absorbe la saturación primero
  Acción: identificar qué operación sube primero (pestaña "Request Details")

7 · Gatling + Kafka — estrategias reales

Gatling en su versión open source es un stress tester HTTP. Para estresar Kafka directamente (sin capa HTTP) hay dos estrategias:

Estrategia A — API REST sobre Kafka (recomendada)

Si tu arquitectura tiene Kafka REST Proxy (Confluent) o un microservicio REST que produce a Kafka, úsalo. Gatling ataca el REST proxy y mide el pipeline completo incluyendo la latencia de producción a Kafka. Es el escenario más realista para Big Data.

KafkaRestProxy.scala — Confluent REST Proxy 
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class KafkaRestProxy extends Simulation {

  val protocolo = http
    .baseUrl("http://kafka-rest-proxy:8082")
    .acceptHeader("application/vnd.kafka.v2+json")
    .contentTypeHeader("application/vnd.kafka.json.v2+json")

  val feeder = Iterator.continually(Map(
    "key"   -> java.util.UUID.randomUUID().toString,
    "value" -> s"""{"event":"click","user":"u${(Math.random()*1000).toInt}"}"""
  ))

  val producir = scenario("Kafka REST Producer")
    .feed(feeder)
    .exec(
      http("Produce a topic")
        .post("/topics/eventos-bigdata")
        .body(StringBody(
          """{"records":[{"key":"#{key}","value":#{value}}]}"""
        ))
        .check(
          status.is(200),
          jsonPath("$.offsets[0].error_code").notExists
        )
    )
    .pause(10.milliseconds, 50.milliseconds)

  setUp(
    producir.inject(
      rampUsers(500).during(30.seconds)
    )
  ).protocols(protocolo)
   .assertions(
     global.responseTime.percentile(95).lte(50),  // Kafka debe ser rápido
     global.failedRequests.percent.lte(0.1)
   )
}

Estrategia B — Plugin Gatling para Kafka

Existe gatling-kafka (librería open source comunitaria) que añade soporte nativo de Kafka al DSL. No es oficial de Gatling pero funciona bien para versiones 3.x.

Dependencia Maven — gatling-kafka 
<dependency>
  <groupId>io.github.amerouseful</groupId>
  <artifactId>gatling-kafka_2.13</artifactId>
  <version>1.1.0</version>
  <scope>test</scope>
</dependency>
KafkaNativeSimulation.scala 
import io.github.amerouseful.gatling.kafka.Predef._
import io.github.amerouseful.gatling.kafka.protocol.KafkaProtocol
import io.gatling.core.Predef._
import org.apache.kafka.clients.producer.ProducerConfig
import scala.concurrent.duration._

class KafkaNativeSimulation extends Simulation {

  val kafkaProtocol: KafkaProtocol = kafka
    .topic("stress-test-topic")
    .properties(Map(
      ProducerConfig.BOOTSTRAP_SERVERS_CONFIG -> "kafka-broker:9092",
      ProducerConfig.ACKS_CONFIG              -> "1",  // acks=1 para máximo throughput
      ProducerConfig.LINGER_MS_CONFIG         -> "5",  // pequeño batching
      ProducerConfig.BATCH_SIZE_CONFIG        -> "16384"
    ))

  val feeder = Iterator.continually(Map(
    "msgKey" -> java.util.UUID.randomUUID().toString,
    "msgVal" -> s"""{"ts":${System.currentTimeMillis()},"sensor":"s${(Math.random()*100).toInt}"}"""
  ))

  val scn = scenario("Kafka Producer Stress")
    .feed(feeder)
    .exec(
      kafka("Enviar evento")
        .send[String, String]
        .key("#{msgKey}")
        .payload("#{msgVal}")
    )
    .pause(5.milliseconds)

  setUp(
    scn.inject(
      rampUsers(1000).during(30.seconds)
    )
  ).protocols(kafkaProtocol)
   .assertions(
     global.failedRequests.percent.lte(1)
   )
}
WARN El plugin gatling-kafka no genera gráficas de latencia por partition en el informe HTML estándar. Para métricas de Kafka a nivel de partition/broker, complementa con JMX exporter de Kafka + Prometheus + Grafana mientras corre el test de Gatling.

8 · Comparativa final — Locust vs Stress Propio vs Gatling

Criterio Locust (Python) StressTester propio (asyncio) Gatling (JVM)
Curva de aprendizaje Baja (Python puro) Media (asyncio) Media (DSL Scala)
Throughput máximo ~10.000 req/s (gevent) ~50.000 req/s (aiohttp) ~100.000 req/s (Netty NIO)
Protocolos HTTP, WebSocket, gRPC* Cualquiera (código) HTTP, WebSocket, gRPC, Kafka*
UI en vivo Sí (web UI en :8089) No (sólo consola) No (informe post-test)
Informes CSV / HTML básico Lo que programes HTML interactivo completo
Integración CI Sí (assertions vía flags) Manual (exit code) Nativa (Maven plugin + assertions)
Modo distribuido Sí (master/worker) No nativo Sí (Gatling Enterprise)
Cuándo usarlo Iteración rápida, laboratorio Protocolos personalizados (Kafka, gRPC directo) Tests de producción, CI, informes formales
SUCCESS La estrategia real: usa Locust para explorar y ajustar durante el desarrollo. Cuando tienes el escenario claro y necesitas saturar de verdad o integrar en CI, migra ese escenario a Gatling. Para Kafka directo o WebSockets con protocolo personalizado, usa el StressTester propio del Cap 9 o el equivalente Java del Cap 11.
