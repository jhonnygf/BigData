El locustfile.py del workspace diseccionado línea a línea. HttpUser, @task, wait_time. Qué significan RPS, p95, p99 y fail_ratio en la UI.

Qué es Locust y cómo funciona internamente

Locust es una herramienta de pruebas de carga escrita en Python. Cada "usuario virtual" es una instancia de una clase HttpUser que ejecuta tareas en bucle. Internamente, Locust usa gevent: una librería de corrutinas basada en greenlets que parchea las librerías de red de Python para que sean no bloqueantes. El resultado es que miles de usuarios virtuales conviven en un solo proceso Python sin crear miles de hilos del SO.

ARQUITECTURA INTERNA DE LOCUST

┌──────────────────────────────────────────────────────────┐
│  Proceso Locust (un solo proceso Python)                 │

│  gevent event loop (basado en libev/libuv)               │

El laboratorio: locustfile.py del workspace

Este es el fichero que existe en el workspace en /Users/dgranados/BigData/Sistemas_de_Big Data/Unidad6_eventos/masivo/locustfile.py. Es el punto de partida mínimo. Lo diseccionamos completo:

python — locustfile.py del workspace (laboratorio real) 
from locust import HttpUser, task, between

class MiPrimerUsuario(HttpUser):
    # Tiempo de espera entre tareas (segundos)
    wait_time = between(1, 3)

    # Host por defecto (podemos no ponerlo y usar --host en CLI)
    host = "http://localhost:8000"

    @task
    def hola(self):
        self.client.get("/hello")

Análisis línea a línea

from locust import HttpUser, task, between

HttpUser es la clase base para usuarios que hacen peticiones HTTP. Incluye un atributo self.client que es una sesión HTTP instrumentada: cada petición que hace se registra automáticamente en las estadísticas de Locust (latencia, estado, fallos).

class MiPrimerUsuario(HttpUser):

Cada instancia de esta clase representa un usuario virtual. Cuando lanzas Locust con 100 usuarios, Locust crea 100 instancias de MiPrimerUsuario, cada una ejecutando sus tareas en su propio greenlet.

wait_time = between(1, 3)

Después de ejecutar cada tarea, el usuario espera un tiempo aleatorio uniforme entre 1 y 3 segundos antes de ejecutar la siguiente. Este tiempo simula el "tiempo de pensar" de un usuario real. Sin wait_time, cada usuario ejecutaría tareas lo más rápido posible, generando una carga irreal.

host = "http://localhost:8000"

El destino de todas las peticiones. Puede sobreescribirse en la CLI con --host sin modificar el código. En el contexto de Big Data, aquí iría la URL del API del clúster: http://kafka-api:8080, http://mongodb-api:5000, etc.

@task

Marca un método como tarea ejecutable. Locust elige aleatoriamente entre los métodos marcados con @task en cada ciclo. Si hay un solo método con @task, siempre ejecuta ese. El decorador acepta un peso entero: @task(3) hace que esa tarea se ejecute 3 veces más que una con @task(1).

self.client.get("/hello")

self.client es una instancia de HttpSession, subclase de requests.Session parchada con gevent. Registra automáticamente: tiempo de respuesta, código HTTP, bytes recibidos, y si hubo error. La ruta "/hello" se concatena con host para formar la URL completa.

La UI de Locust: qué leer y qué significa

PANTALLA DE CONFIGURACIÓN INICIAL (http://localhost:8089)

┌────────────────────────────────────────────────────┐
│  Number of users (peak concurrency): [100]         │
│  Spawn rate (users started/second): [10]           │
│  Host: http://localhost:8000                       │
│                                         [Start]   │
└────────────────────────────────────────────────────┘

Number of users: cuántos usuarios virtuales habrá en total al final del ramp-up
Spawn rate: cuántos usuarios nuevos se crean por segundo hasta llegar al total
  → con 100 usuarios y spawn rate 10: los 100 usuarios estarán activos en 10 segundos
      
PANTALLA DE ESTADÍSTICAS EN TIEMPO REAL

┌────────────────────────────────────────────────────────────────────────────┐
│ TYPE  NAME          # Reqs  # Fails  Avg(ms)  Min  Max  Avg size  RPS     │
│ GET   /hello         4,823      12     234    102  891    1,024   48.2   │
│ ─────────────────────────────────────────────────────────────────────────  │
│ Aggregated           4,823      12     234    102  891    1,024   48.2   │
└────────────────────────────────────────────────────────────────────────────┘

# Reqs:     peticiones totales realizadas desde el inicio del test
# Fails:    peticiones que devolvieron error (4xx, 5xx, timeout, excepción)
Avg(ms):    latencia media — puede ser engañosa si hay outliers
Min/Max:    latencia mínima y máxima observadas
RPS:        requests per second en el momento actual
      
PESTAÑA "CHARTS" — métricas en tiempo real

RPS chart:    muestra cómo sube el throughput durante el ramp-up y si se estabiliza
Response time chart:
  - Línea verde:  p50 (mediana) — la mitad de las peticiones son más rápidas que esto
  - Línea naranja: p95 — el 95% de las peticiones son más rápidas que esto
  - Línea roja:   p99 — el 99% de las peticiones son más rápidas que esto

Users chart:  muestra cómo crecen los usuarios durante el ramp-up

PESTAÑA "FAILURES" — errores con detalle
  Muestra las excepciones y sus conteos. Aquí verás:
  - ConnectionRefusedError: el servidor no acepta más conexiones
  - HTTPError 500: el servidor está fallando internamente
  - ReadTimeout: el servidor no responde en tiempo

Métricas críticas: p95 y p99

La latencia media (Avg) es la métrica más usada y la más engañosa. Si 95 peticiones tardan 100ms y 5 tardan 5 segundos, la media es ~350ms pero la experiencia real de 1 de cada 20 usuarios es terrible.

Los percentiles dan una visión honesta de la distribución:

p50

Mediana

La petición "típica". La mitad van más rápido, la mitad más lento. El usuario medio experimenta esto.

p95

Percentil 95

El 95% de las peticiones son más rápidas que esto. El 5% más lento experimenta tiempos mayores.

p99

Percentil 99

El 1% más lento. En un sistema con 10.000 req/s, ese 1% son 100 peticiones por segundo con latencia alta.

python — calcular percentiles manualmente para entender la distribución 
import statistics

latencias = [100, 105, 98, 102, 110, 95, 103, 108, 99, 101,
             4500, 3200, 5100, 4800, 3900]  # 5 peticiones lentas

latencias.sort()
n = len(latencias)

media    = statistics.mean(latencias)
mediana  = statistics.median(latencias)
p95      = latencias[int(n * 0.95)]
p99      = latencias[int(n * 0.99)]

print(f"Media:   {media:.0f}ms  ← engañosa (inflada por outliers)")
print(f"Mediana: {mediana:.0f}ms  ← la experiencia real del usuario típico")
print(f"p95:     {p95:.0f}ms")
print(f"p99:     {p99:.0f}ms")

Ampliar el locustfile del workspace: varias tareas con pesos

El locustfile original tiene una sola tarea. En un entorno de Big Data real, un clúster recibe diferentes tipos de peticiones con diferentes frecuencias. Aquí ampliamos el locustfile para simular un patrón más realista: lecturas frecuentes + escrituras ocasionales.

python — locustfile ampliado: simulación realista de carga Big Data 
from locust import HttpUser, task, between
import random

class BigDataUser(HttpUser):
    wait_time = between(0.5, 2)
    host = "http://localhost:8000"

    @task(5)  # lectura: 5x más frecuente que escritura
    def leer_datos(self):
        """Simula una consulta de lectura al clúster."""
        limite = random.choice([10, 50, 100, 500])
        self.client.get(
            f"/api/datos?limit={limite}&sort=-timestamp",
            name="GET /api/datos"  # nombre en UI independiente de los params
        )

    @task(2)  # inserción batch: 2x más frecuente que bulk
    def insertar_registro(self):
        """Simula inserción de un registro."""
        payload = {
            "timestamp": "2026-04-27T10:00:00Z",
            "sensor_id": random.randint(1, 100),
            "valor": random.uniform(-50, 150)
        }
        self.client.post(
            "/api/datos",
            json=payload,
            name="POST /api/datos"
        )

    @task(1)  # bulk insert: menos frecuente pero más pesado
    def bulk_insert(self):
        """Simula inserción masiva de 500 registros."""
        documentos = [
            {"sensor_id": random.randint(1, 100), "valor": random.random()}
            for _ in range(500)
        ]
        self.client.post(
            "/api/bulk",
            json={"documentos": documentos},
            name="POST /api/bulk (500 docs)"
        )
INFO
El parámetro name en las peticiones: Sin el parámetro name, Locust agrupa las estadísticas por URL exacta. Con parámetros variables como /api/datos?limit=10, /api/datos?limit=50, etc., verías decenas de filas en la UI, una por cada combinación. Con name="GET /api/datos", todas van a la misma fila.

Ejecutar el test y verificar el sistema bajo prueba

bash — lanzar Locust y monitorizar el sistema al mismo tiempo 
# Terminal 1: lanzar Locust
locust -f locustfile.py --host http://localhost:8000

# Terminal 2: monitorizar CPU y RAM del proceso destino mientras corre el test
# Reemplaza 8000 con el puerto real de tu sistema
watch -n 1 'lsof -i :8000 | head -5; echo "---"; ps aux | grep -v grep | grep -E "mongo|kafka|python|java" | awk "{print \$1,\$3,\$4,\$11}"'

# Terminal 3: si el sistema es MongoDB, monitorizar métricas en tiempo real
mongosh --eval "db.serverStatus().opcounters"
# o con mongotop:
mongotop 1  # muestra reads/writes por colección cada 1 segundo
WARN
Siempre monitoriza el servidor durante el test. Si la CPU del servidor sube al 100%, el cuello de botella está en el servidor. Si la CPU del servidor está al 10% y los RPS no suben, el cuello de botella puede ser la red, la configuración de conexiones máximas, o el propio generador de carga. No confundas "el test va lento" con "el servidor no aguanta".
