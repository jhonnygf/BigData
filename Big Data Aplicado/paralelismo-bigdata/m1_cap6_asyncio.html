asyncio: I/O masivo con un solo hilo

El event loop, corrutinas, gather, create_task y aiohttp. Por qué asyncio puede abrir 10.000 conexiones donde threading colapsa con 500.

El modelo mental: un solo hilo que nunca bloquea

Threading resuelve el problema de I/O concurrente creando un hilo por cada tarea en espera. Funciona, pero tiene un coste: cada hilo del SO consume ~8MB de stack, requiere sincronización entre sí y el scheduler del SO tiene que hacer context switch entre ellos. Con 500 hilos, ese overhead se nota.

asyncio resuelve el mismo problema con un solo hilo y un bucle de eventos (event loop). El event loop monitoriza todas las operaciones de I/O pendientes usando las primitivas del SO (select/epoll/kqueue). Cuando una operación completa, el event loop despierta la corrutina que la estaba esperando y la ejecuta hasta que vuelve a hacer await sobre algo.

EVENT LOOP (un solo hilo, sin GIL contención)

          ┌─────────────────────────────────────────┐
          │            EVENT LOOP                   │
          │                                         │
          │  Corrutina A: await red   (durmiendo)  │
          │  Corrutina B: await BD    (durmiendo)  │
          │  Corrutina C: ejecutando  (activa)     │◄── solo una activa a la vez
          │  Corrutina D: await disco (durmiendo)  │
          │  ...10.000 corrutinas más...            │
          │                                         │
          │  epoll/kqueue: avisa cuando I/O termina │
          └────────────────┬────────────────────────┘
                           │
                     SO notifica:
                  "red de Corrutina A lista"
                           │
                           ▼
          Corrutina A despierta, ejecuta hasta el próximo await
INFO
Por qué escala tanto: 10.000 corrutinas durmiendo en await cuestan unos pocos KB cada una (solo el estado de la corrutina en heap). 10.000 hilos durmiendo cuestan ~80GB de stack. El event loop nunca tiene que hacer context switch del SO entre corrutinas — el switch lo hace el código Python en cada await.

Corrutinas: funciones que saben pausarse

Una corrutina es una función definida con async def. Cuando llega a un await, cede el control al event loop. El event loop puede ejecutar otras corrutinas mientras esta espera. Cuando la operación awaited termina, el event loop devuelve el control a esta corrutina desde donde se quedó.

python — primera corrutina: async def y await 
import asyncio
import time

async def tarea(nombre, segundos):
    print(f"[{nombre}] Inicio. Esperará {segundos}s")
    await asyncio.sleep(segundos)  # cede control; el event loop ejecuta otras corrutinas
    print(f"[{nombre}] Terminado")
    return f"resultado_{nombre}"

async def main():
    t0 = time.perf_counter()

    # Ejecutar las 3 tareas de forma SECUENCIAL (una esperamos antes de llamar a la siguiente)
    r1 = await tarea("A", 1)
    r2 = await tarea("B", 1)
    r3 = await tarea("C", 1)
    print(f"Secuencial: {time.perf_counter()-t0:.2f}s → {r1}, {r2}, {r3}")

asyncio.run(main())

Output (secuencial, cada tarea espera a la anterior):

output — secuencial
[A] Inicio. Esperará 1s
[A] Terminado
[B] Inicio. Esperará 1s
[B] Terminado
[C] Inicio. Esperará 1s
[C] Terminado
Secuencial: 3.01s → resultado_A, resultado_B, resultado_C
WARN
Trampa clásica: await tarea_A() seguido de await tarea_B() es SECUENCIAL, no concurrente. Para ejecutar en paralelo hay que usar gather() o create_task().

gather(): lanzar múltiples corrutinas en paralelo

asyncio.gather(*corrutinas) lanza todas las corrutinas al mismo tiempo y espera a que todas terminen. Devuelve una lista de resultados en el mismo orden que las corrutinas pasadas.

python — gather: 3 tareas en paralelo 
import asyncio
import time

async def tarea(nombre, segundos):
    print(f"[{nombre}] Inicio")
    await asyncio.sleep(segundos)
    print(f"[{nombre}] Terminado")
    return f"resultado_{nombre}"

async def main():
    t0 = time.perf_counter()

    # Ejecutar las 3 tareas CONCURRENTEMENTE
    resultados = await asyncio.gather(
        tarea("A", 1),
        tarea("B", 1),
        tarea("C", 1),
    )
    print(f"gather: {time.perf_counter()-t0:.2f}s → {resultados}")

asyncio.run(main())

Output (concurrente: las 3 arrancan a la vez):

output — concurrent con gather
[A] Inicio
[B] Inicio
[C] Inicio
[A] Terminado
[B] Terminado
[C] Terminado
gather: 1.01s → ['resultado_A', 'resultado_B', 'resultado_C']

create_task(): control fino sobre tareas individuales

asyncio.create_task(corrutina) programa la corrutina para ejecutarse en el event loop y devuelve un Task inmediatamente. El Task comienza a ejecutarse en el background en cuanto hay un await en el código actual que cede el control.

python — create_task: procesar resultados según llegan 
import asyncio
import time
import random

async def fetch_dato(id):
    latencia = random.uniform(0.1, 1.5)
    await asyncio.sleep(latencia)
    return {"id": id, "latencia_ms": latencia * 1000}

async def main():
    t0 = time.perf_counter()

    # Crear 20 tasks de golpe
    tasks = [asyncio.create_task(fetch_dato(i)) for i in range(20)]

    # asyncio.as_completed: procesar según van terminando
    for coro in asyncio.as_completed(tasks):
        resultado = await coro
        elapsed = time.perf_counter() - t0
        print(f"t+{elapsed:.2f}s: dato {resultado['id']} → {resultado['latencia_ms']:.0f}ms")

    print(f"\nTotal: {time.perf_counter()-t0:.2f}s (secuencial hubiera tardado ~{sum(random.uniform(0.1,1.5) for _ in range(20)):.1f}s)")

asyncio.run(main())

aiohttp: peticiones HTTP con asyncio

aiohttp es la librería HTTP asíncrona estándar para asyncio. Su ClientSession gestiona el pool de conexiones. Es la base del stress tester que construiremos en el Capítulo 9.

pip install aiohttp
# o con conda:
conda install -c conda-forge aiohttp
python — aiohttp: 100 peticiones HTTP en paralelo 
import asyncio
import aiohttp
import time

async def fetch(session, url, semaforo):
    async with semaforo:  # limita concurrencia máxima
        t0 = time.perf_counter()
        async with session.get(url) as resp:
            await resp.read()
            return {
                "url": url,
                "status": resp.status,
                "latencia_ms": (time.perf_counter() - t0) * 1000
            }

async def main():
    N = 100
    MAX_CONCURRENT = 50  # no más de 50 conexiones simultáneas al servidor
    URL = "https://httpbin.org/status/200"

    semaforo = asyncio.Semaphore(MAX_CONCURRENT)

    t0 = time.perf_counter()
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, URL, semaforo) for _ in range(N)]
        resultados = await asyncio.gather(*tasks, return_exceptions=True)

    elapsed = time.perf_counter() - t0
    ok = [r for r in resultados if isinstance(r, dict) and r["status"] == 200]
    errores = [r for r in resultados if isinstance(r, Exception)]
    latencias = [r["latencia_ms"] for r in ok]

    print(f"N={N} peticiones en {elapsed:.2f}s → {N/elapsed:.1f} req/s")
    print(f"OK: {len(ok)} | Errores: {len(errores)}")
    if latencias:
        latencias.sort()
        print(f"Latencia — p50: {latencias[len(latencias)//2]:.0f}ms | p95: {latencias[int(len(latencias)*0.95)]:.0f}ms")

asyncio.run(main())
INFO
Semaphore en asyncio: asyncio.Semaphore(N) limita cuántas corrutinas pueden estar en la sección crítica a la vez. Esencial para no saturar al servidor destino con más conexiones de las que puede manejar. Sin él, las 100 peticiones intentan conectar exactamente al mismo instante.

Comparativa observable: threading vs asyncio a escala

python — threading vs asyncio con 500 peticiones 
import asyncio
import aiohttp
import threading
import urllib.request
import time

URL = "https://httpbin.org/status/200"
N = 200  # ajusta según tu conexión

# ── THREADING ────────────────────────────────────────────────────────────────
def fetch_sync(url):
    try:
        with urllib.request.urlopen(url, timeout=10): pass
    except: pass

t0 = time.perf_counter()
hilos = [threading.Thread(target=fetch_sync, args=(URL,)) for _ in range(N)]
for h in hilos: h.start()
for h in hilos: h.join()
t_threading = time.perf_counter() - t0
print(f"Threading ({N} hilos):  {t_threading:.2f}s → {N/t_threading:.1f} req/s")

# ── ASYNCIO ───────────────────────────────────────────────────────────────────
async def fetch_async(session, sem):
    async with sem:
        async with session.get(URL) as r:
            await r.read()

async def benchmark_asyncio():
    sem = asyncio.Semaphore(N)  # sin límite adicional: los N a la vez
    async with aiohttp.ClientSession() as s:
        await asyncio.gather(*[fetch_async(s, sem) for _ in range(N)])

t0 = time.perf_counter()
asyncio.run(benchmark_asyncio())
t_asyncio = time.perf_counter() - t0
print(f"asyncio  ({N} corrutinas): {t_asyncio:.2f}s → {N/t_asyncio:.1f} req/s")

Output típico (la diferencia se amplía con N mayor):

output
Threading (200 hilos):    8.3s → 24.1 req/s
asyncio  (200 corrutinas): 3.1s → 64.5 req/s

Bloqueo accidental: la trampa más común

Si llamas a una función bloqueante (que no es async) desde dentro de una corrutina, bloqueas el event loop entero. Mientras esa función no retorna, ninguna otra corrutina puede ejecutar.

python — bloqueo accidental vs solución correcta 
import asyncio
import time
import urllib.request
import aiohttp

# MAL: urllib es bloqueante. Bloquea el event loop entero durante la petición
async def fetch_mal(url):
    with urllib.request.urlopen(url) as r:  # BLOQUEA el event loop
        return r.read()

# BIEN: usar aiohttp (no bloqueante)
async def fetch_bien(session, url):
    async with session.get(url) as r:  # cede el control mientras espera
        return await r.read()

# ALTERNATIVA: ejecutar código bloqueante en un thread pool para no bloquear el loop
async def fetch_con_executor(url):
    loop = asyncio.get_event_loop()
    # run_in_executor envía la función bloqueante a un ThreadPoolExecutor
    return await loop.run_in_executor(None, urllib.request.urlopen, url)
FAIL
Síntoma de bloqueo accidental: El event loop tiene 1000 corrutinas pero solo procesa ~1 petición por vez. El throughput es idéntico al secuencial. Esto pasa cuando código bloqueante (requests, psycopg2, time.sleep, fichero síncrono) se llama sin await desde dentro de una corrutina.
