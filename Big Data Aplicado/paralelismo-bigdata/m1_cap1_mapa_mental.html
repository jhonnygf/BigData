Módulo 1 — Capítulo 1

El mapa mental: concurrencia, paralelismo y asincronía

El diagnóstico previo que decide qué herramienta usar. Elegir mal aquí hace que el código sea más lento que la versión secuencial.

Por qué este capítulo es el más importante

El error más frecuente al trabajar con sistemas de alto rendimiento es elegir la herramienta de paralelismo sin entender primero qué tipo de trabajo ejecuta el programa. Un programa que hace miles de peticiones HTTP mejora dramáticamente con asyncio. Ese mismo código con multiprocessing puede ser un 30% más lento que la versión secuencial, por el overhead de crear procesos.

Antes de escribir una sola línea de código paralelo, hay que responder una pregunta: ¿mi programa está esperando la CPU o está esperando algo externo?

INFO
Regla de diagnóstico: Si el programa consume CPU al 100% mientras trabaja, es CPU-bound. Si el proceso duerme esperando red, disco o base de datos, es IO-bound. Esta distinción define completamente qué herramienta funciona.

Los tres conceptos que se confunden siempre

CONCURRENCIA

Concurrencia

Varias tareas en progreso al mismo tiempo, pero no necesariamente ejecutándose en el mismo instante. Una sola CPU puede ser concurrente alternando entre tareas (context switch).

PARALELISMO

Paralelismo

Varias tareas ejecutándose literalmente al mismo tiempo en múltiples núcleos de CPU. Requiere múltiples unidades de ejecución físicas.

ASINCRONÍA

Asincronía

Una tarea lanza una operación y no se bloquea esperando el resultado. Continúa con otra tarea. Cuando llega el resultado, retoma. No implica múltiples hilos ni procesos.

La confusión habitual: pensar que asincronía = paralelismo. No lo es. asyncio en Python es concurrente pero no paralelo: hay un solo hilo de ejecución. Lo que gana es no desperdiciar tiempo de CPU bloqueado esperando I/O.

El modelo de ejecución interno: qué ocurre en la CPU

Un proceso de Python tiene, por defecto, un único hilo de ejecución. Ese hilo corre en un núcleo de CPU. Cuando el hilo llega a una instrucción de red o disco (una llamada al sistema operativo), ocurre esto:

INFO
Conclusión del diagrama: La concurrencia gana cuando el tiempo de espera (I/O) domina. El paralelismo gana cuando el tiempo de CPU domina. Si mezclas los casos, obtienes el peor resultado de ambos mundos.

CPU-bound vs IO-bound: el diagnóstico real

CPU-bound: el programa trabaja

Un programa CPU-bound consume ciclos de procesador sin detenerse. Ejemplos reales en Big Data:

  • Calcular una transformación sobre 50 millones de registros (map/reduce manual)
  • Entrenar un modelo de ML en CPU
  • Serializar/deserializar JSON de alto volumen (sí, parsear JSON es CPU-bound)
  • Comprimir o descomprimir un fichero grande con gzip/zstd
  • Calcular hashes criptográficos sobre un dataset

Para este tipo de trabajo, la solución es multiprocessing: múltiples procesos Python, cada uno con su intérprete y su GIL propio, corriendo en núcleos distintos. Threading no ayuda aquí por culpa del GIL (ver Capítulo 2).

IO-bound: el programa espera

Un programa IO-bound pasa la mayor parte del tiempo bloqueado esperando una respuesta externa. Ejemplos en Big Data:

  • Hacer 10.000 peticiones HTTP a un API REST para insertar datos
  • Leer 500 ficheros de S3 o HDFS de forma secuencial
  • Consultar MongoDB con 1000 queries distintas
  • Publicar mensajes en Kafka esperando el ACK de cada uno
  • Abrir conexiones TCP a múltiples nodos de un clúster

Para este trabajo, tanto threading como asyncio funcionan. La diferencia entre ellos está en la escala: threading aguanta cientos de conexiones simultáneas; asyncio aguanta decenas de miles.

Cómo medir si tu programa es CPU-bound o IO-bound

Antes de escribir código paralelo, mide. Hay dos formas rápidas:

Opción 1: observar el monitor de sistema

bash — observar uso de CPU durante ejecución 
# En una terminal, lanza el programa
python mi_script.py &

# En otra terminal, observa en tiempo real
# Si el % CPU se mantiene cerca de 100% durante toda la ejecución: CPU-bound
# Si el % CPU oscila cerca de 0-10%: IO-bound
top -pid $(pgrep -f mi_script)

# Alternativa con htop (más visual)
htop

Opción 2: usar cProfile para ver dónde pasa el tiempo

bash — profiling de un script 
python -m cProfile -s cumulative mi_script.py | head -30

# Busca en la salida:
# - Funciones como select(), recv(), send(), sleep() con mucho tiempo acumulado → IO-bound
# - Funciones de cálculo propio con mucho tiempo acumulado → CPU-bound
WARN
Trampa frecuente: Un programa que hace muchas peticiones HTTP de forma secuencial aparece como CPU-bound en top porque el proceso nunca duerme más de un instante — siempre está lanzando la siguiente petición. El síntoma real es que el tiempo de ejecución baja drásticamente al paralelizarlo, lo que confirma que el cuello de botella era la espera, no la CPU.

El mapa de decisión: qué herramienta usar

¿Qué tipo de trabajo hace mi programa?
│
├── IO-bound (espera red, disco, DB, APIs)
│   │
│   ├── ¿Menos de 500 tareas simultáneas y el código ya usa threads?

│   │

│   └── ¿Quiero hacer stress testing HTTP?

    ├── ¿Quiero usar Python?

        └── Java  →  Fork/Join, CompletableFuture  (Cap 10 y 11)

Un experimento para ver la diferencia en números reales

El siguiente script ejecuta la misma carga de trabajo de tres formas distintas y mide el tiempo. Cópialo, ejecútalo y observa los resultados. El objetivo es ver empíricamente que la elección de herramienta importa más que cualquier optimización de código.

python — comparativa secuencial vs threading vs multiprocessing 
import time
import threading
import multiprocessing
import urllib.request

# ── TAREA CPU-BOUND ──────────────────────────────────────────────────────────
def tarea_cpu(n):
    """Calcula la suma de cuadrados hasta n: puro trabajo de CPU."""
    return sum(i * i for i in range(n))

# ── TAREA IO-BOUND ───────────────────────────────────────────────────────────
def tarea_io(url):
    """Descarga una URL: trabajo de red, el proceso duerme esperando."""
    try:
        with urllib.request.urlopen(url, timeout=5) as r:
            return len(r.read())
    except Exception:
        return 0

# ── BENCHMARK CPU-BOUND ──────────────────────────────────────────────────────
N = 5_000_000
WORKERS = 4

print("=" * 60)
print("BENCHMARK CPU-BOUND (suma de cuadrados hasta 5M)")
print("=" * 60)

# Secuencial
t0 = time.perf_counter()
for _ in range(WORKERS):
    tarea_cpu(N)
print(f"Secuencial:      {time.perf_counter() - t0:.2f}s")

# Threading (el GIL lo limita — verás que no mejora)
t0 = time.perf_counter()
hilos = [threading.Thread(target=tarea_cpu, args=(N,)) for _ in range(WORKERS)]
for h in hilos: h.start()
for h in hilos: h.join()
print(f"Threading:       {time.perf_counter() - t0:.2f}s  ← igual o peor (GIL)")

# Multiprocessing (procesos reales, sin GIL compartido)
t0 = time.perf_counter()
with multiprocessing.Pool(WORKERS) as pool:
    pool.map(tarea_cpu, [N] * WORKERS)
print(f"Multiprocessing: {time.perf_counter() - t0:.2f}s  ← debería ser ~{1/WORKERS:.1f}x más rápido")

# ── BENCHMARK IO-BOUND ───────────────────────────────────────────────────────
URLS = ["http://httpbin.org/delay/1"] * 8  # 8 URLs, cada una tarda ~1s

print()
print("=" * 60)
print("BENCHMARK IO-BOUND (8 peticiones HTTP con 1s de delay cada una)")
print("=" * 60)

# Secuencial (esperamos cada respuesta antes de pedir la siguiente)
t0 = time.perf_counter()
for url in URLS:
    tarea_io(url)
print(f"Secuencial: {time.perf_counter() - t0:.1f}s  ← ~8 segundos (1s × 8)")

# Threading (mientras una espera, otra trabaja)
t0 = time.perf_counter()
hilos = [threading.Thread(target=tarea_io, args=(url,)) for url in URLS]
for h in hilos: h.start()
for h in hilos: h.join()
print(f"Threading:  {time.perf_counter() - t0:.1f}s  ← ~1 segundo (todas en paralelo)")

Output esperado (en una máquina con 4 núcleos):

output — resultado del benchmark 
============================================================
BENCHMARK CPU-BOUND (suma de cuadrados hasta 5M)
============================================================
Secuencial:      4.12s
Threading:       4.18s  ← igual o peor (GIL)
Multiprocessing: 1.09s  ← debería ser ~0.2x más rápido

============================================================
BENCHMARK IO-BOUND (8 peticiones HTTP con 1s de delay cada una)
============================================================
Secuencial: 8.3s  ← ~8 segundos (1s × 8)
Threading:  1.1s  ← ~1 segundo (todas en paralelo)
INFO
Qué confirma este experimento: Para trabajo CPU-bound, threading no ayuda (incluso perjudica ligeramente por el overhead). Multiprocessing sí escala. Para trabajo IO-bound, threading sí escala porque el GIL se libera durante las llamadas al sistema operativo. Este comportamiento se explica en detalle en el Capítulo 2.

El caso especial del stress testing

Cuando el objetivo es generar carga sobre un sistema externo (un API REST, un topic Kafka, un endpoint de MongoDB), el programa generador de carga es por definición IO-bound: lanza peticiones y espera respuestas.

Esto tiene una implicación directa: una sola máquina puede generar una cantidad enorme de carga si usa las herramientas correctas. Con threading puro, una máquina puede sostener ~500-1000 conexiones simultáneas sin problemas. Con asyncio (o con Locust, que usa gevent internamente), la misma máquina puede sostener 10.000-50.000 "usuarios virtuales".

Cuando ni una máquina alcanza, se distribuye la carga: varios procesos Locust en varias máquinas apuntando al mismo sistema objetivo. Esto lo vemos en el Capítulo 8.

WARN
Límite real de la máquina generadora: Si el generador de carga es el cuello de botella (CPU al 100%, o memoria agotada), los resultados del test no reflejan la capacidad del sistema bajo prueba, sino la capacidad del generador. Siempre monitoriza la máquina generadora durante el test.

Resumen del capítulo

multiprocessing CPU-bound Tantos como núcleos físicos IO-bound (overhead de procesos)
