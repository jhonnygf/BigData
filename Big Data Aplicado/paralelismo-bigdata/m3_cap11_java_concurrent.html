Módulo 3 — Java/JVM · Capítulo 11

java.util.concurrent

CountDownLatch · CyclicBarrier · Semaphore · CompletableFuture · Fork/Join

El arsenal real de concurrencia en Java

El paquete java.util.concurrent (JUC) fue introducido en Java 5 y es la razón por la que Java sigue siendo el lenguaje de referencia para Big Data distribuido. Spark, Kafka, Flink, HBase: todos construyen sobre estas primitivas. En el capítulo anterior cubrimos los bloques base: Thread, ExecutorService, synchronized, AtomicInteger, BlockingQueue. Aquí subimos un peldaño.

Este capítulo cubre las herramientas de coordinación entre hilos (no sólo ejecución), la programación reactiva con CompletableFuture, y el modelo Fork/Join para dividir trabajo CPU-bound masivo. Todo orientado a construir herramientas de stress testing que puedan saturar un clúster Kafka o Spark sin que el generador sea el cuello de botella.

CountDownLatch

Barrera de cuenta regresiva. N hilos decrementan; el que espera avanza cuando llega a 0. Un disparo, no recargable.

CyclicBarrier

Punto de encuentro: todos esperan hasta que el último llega. Recargable: puede usarse en bucle para fases.

Semaphore

Control de acceso concurrente. Limita cuántos hilos pueden estar en una sección al mismo tiempo.

CompletableFuture

Pipeline reactivo asíncrono. Encadena operaciones sin bloquear: thenApply, thenCompose, allOf, exceptionally.

Fork/Join Framework

Divide y vencerás a escala JVM. Work-stealing para procesar grandes volúmenes de datos con todos los cores.

1 · CountDownLatch — arranque coordinado

Mecanismo

CountDownLatch se inicializa con un contador entero N. Cualquier hilo puede llamar a countDown() para decrementarlo. El hilo que llama a await() se bloquea hasta que el contador llega a cero. Una vez a cero, todos los hilos bloqueados en await() se liberan simultáneamente. El latch no se puede reiniciar (a diferencia de CyclicBarrier). 

  Hilo principal
       │
       │  new CountDownLatch(3)
       │         │
       │    ┌────┴────────────────────────┐
       │    │ Worker-1                    │  countDown()  →  latch = 2
       │    │ Worker-2                    │  countDown()  →  latch = 1
       │    │ Worker-3                    │  countDown()  →  latch = 0
       │    └────────────────────────────┘
       │
   await()  ←  desbloquea cuando latch = 0
       │
  [continúa el trabajo post-inicio]
WARN CountDownLatch no se puede reutilizar. Si necesitas barreras en bucle para múltiples fases usa CyclicBarrier o crea un nuevo latch cada vez.

Ejemplo base — señal de inicio

Caso clásico: 100 trabajadores se preparan. El coordinador da la señal de inicio; todos arrancan exactamente al mismo tiempo. Esto simula un burst de carga coordinado — el patrón exacto que usamos en stress testing.

CoordStart.java 
import java.util.concurrent.*;

public class CoordStart {

    public static void main(String[] args) throws InterruptedException {
        int WORKERS = 10;

        // Un latch para "todos listos" (workers decrementan)
        CountDownLatch listos = new CountDownLatch(WORKERS);
        // Un latch para "arrancar" (el principal lo abre)
        CountDownLatch arranque = new CountDownLatch(1);

        ExecutorService pool = Executors.newFixedThreadPool(WORKERS);

        for (int i = 0; i < WORKERS; i++) {
            final int id = i;
            pool.submit(() -> {
                System.out.printf("[W%02d] Preparado%n", id);
                listos.countDown();         // notifica que está listo
                try {
                    arranque.await();       // espera la señal
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return;
                }
                System.out.printf("[W%02d] DISPARANDO peticiones%n", id);
                // Aquí irían las peticiones HTTP/Kafka
            });
        }

        listos.await();             // espera a que todos estén listos
        System.out.println(">>> TODOS LISTOS — ARRANCANDO");
        arranque.countDown();       // señal de disparo simultáneo

        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);
    }
}

Salida esperada

[W03] Preparado
[W07] Preparado
[W01] Preparado
...
>>> TODOS LISTOS — ARRANCANDO
[W03] DISPARANDO peticiones
[W07] DISPARANDO peticiones
[W01] DISPARANDO peticiones

Ejemplo real — medir tiempo de arranque de Kafka

Variante útil para stress: medir cuánto tarda el sistema en responder al primer burst. Todos los productores Kafka arrancan en el mismo instante; registramos el tiempo de la primera respuesta.

KafkaBurstTimer.java 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class KafkaBurstTimer {

    static final int PRODUCERS = 50;
    static final AtomicLong primeraRespuesta = new AtomicLong(Long.MAX_VALUE);

    public static void main(String[] args) throws InterruptedException {
        CountDownLatch listos  = new CountDownLatch(PRODUCERS);
        CountDownLatch disparo = new CountDownLatch(1);

        ExecutorService pool = Executors.newFixedThreadPool(PRODUCERS);
        long[] tiempoInicio = new long[1];

        for (int i = 0; i < PRODUCERS; i++) {
            pool.submit(() -> {
                listos.countDown();
                try { disparo.await(); } catch (InterruptedException e) { return; }

                long t0 = System.nanoTime();
                // simulamos latencia de envío Kafka
                try { Thread.sleep((long)(Math.random() * 20)); }
                catch (InterruptedException ignored) {}
                long latencia = System.nanoTime() - t0;

                // guardamos la latencia más baja (primera respuesta)
                primeraRespuesta.updateAndGet(v -> Math.min(v, latencia));
            });
        }

        listos.await();
        tiempoInicio[0] = System.nanoTime();
        disparo.countDown();  // disparo simultáneo

        pool.shutdown();
        pool.awaitTermination(30, TimeUnit.SECONDS);

        System.out.printf("Primera respuesta (de %d productores simultáneos): %.2f ms%n",
            PRODUCERS, primeraRespuesta.get() / 1_000_000.0);
        System.out.printf("Tiempo total burst: %.2f ms%n",
            (System.nanoTime() - tiempoInicio[0]) / 1_000_000.0);
    }
}
INFO El doble-latch (listos + disparo) es el patrón estándar de stress testing para simular carga simultánea real. Sin el latch de disparo, los workers arrancan escalonados según son creados por el thread pool — no hay burst real.

2 · CyclicBarrier — fases sincronizadas

Mecanismo

CyclicBarrier funciona como un punto de encuentro: todos los hilos que llaman a await() se bloquean hasta que el último llega. En ese momento, todos avanzan juntos. La diferencia clave con CountDownLatch: la barrera se reinicia automáticamente después de cada ciclo, lo que permite múltiples fases.

Adicionalmente, acepta una acción de barrera (Runnable) que se ejecuta una vez, por el último hilo en llegar, antes de liberar a todos los demás. 

  Fase 1                     Fase 2                     Fase 3
  ─────────────────────────  ─────────────────────────  ──────
  W1 ──────────┐             W1 ──────────┐             W1 ──►
  W2 ──────────┤  [barrera]  W2 ──────────┤  [barrera]  W2 ──►
  W3 ──────────┤  ─────────► W3 ──────────┤  ─────────► W3 ──►
  W4 ──────────┘             W4 ──────────┘             W4 ──►
                 ↑ acción de barrera (resumen de fase)

Ejemplo base — pipeline de carga por fases

En stress testing masivo útil para simular: fase WARM-UP → fase RAMP-UP → fase PICO → fase COOL-DOWN. Cada hilo completa su trabajo en la fase actual y espera al resto antes de pasar a la siguiente.

FasesStress.java 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class FasesStress {

    static final String[] FASES = {"WARM-UP", "RAMP-UP", "PICO", "COOL-DOWN"};
    static final AtomicInteger faseActual = new AtomicInteger(0);
    static final AtomicLong totalPeticiones = new AtomicLong(0);

    public static void main(String[] args) throws InterruptedException {
        int WORKERS = 8;
        int PETICIONES_POR_FASE = 100;

        // Acción de barrera: se ejecuta cuando todos los workers terminan la fase
        Runnable accionBarrera = () -> {
            int fase = faseActual.getAndIncrement();
            System.out.printf("%n═══ FIN %s — total acumulado: %d peticiones ═══%n",
                FASES[fase], totalPeticiones.get());
        };

        CyclicBarrier barrera = new CyclicBarrier(WORKERS, accionBarrera);
        ExecutorService pool   = Executors.newFixedThreadPool(WORKERS);

        for (int w = 0; w < WORKERS; w++) {
            final int wid = w;
            pool.submit(() -> {
                try {
                    for (String fase : FASES) {
                        // trabajo de esta fase
                        int peticiones = 0;
                        for (int p = 0; p < PETICIONES_POR_FASE; p++) {
                            Thread.sleep(1);  // simula latencia
                            peticiones++;
                        }
                        totalPeticiones.addAndGet(peticiones);
                        System.out.printf("  [W%02d] %s completada (%d pet.)%n",
                            wid, fase, peticiones);

                        barrera.await();  // espera al resto antes de pasar de fase
                    }
                } catch (InterruptedException | BrokenBarrierException e) {
                    System.err.println("Worker " + wid + " interrumpido: " + e.getMessage());
                }
            });
        }

        pool.shutdown();
        pool.awaitTermination(60, TimeUnit.SECONDS);
    }
}

Caso de fallo — BrokenBarrierException

FAIL Si cualquier hilo que está esperando en la barrera es interrumpido o lanza excepción, la barrera queda "rota" (BrokenBarrierException) y todos los hilos que estén o lleguen a await() reciben la excepción. Hay que manejarla para no dejar hilos colgados indefinidamente.
BrokenBarrierDemo.java (patrón defensivo) 
try {
    barrera.await(5, TimeUnit.SECONDS);  // timeout: no esperar para siempre
} catch (TimeoutException e) {
    System.err.println("Timeout en barrera — algún worker colgado");
    barrera.reset();  // reinicia para recuperación
} catch (BrokenBarrierException e) {
    System.err.println("Barrera rota — otro worker falló");
    // abortar o reintentar según la lógica del test
}

3 · Semaphore — control de concurrencia

Mecanismo

Un Semaphore mantiene un conjunto de permisos. acquire() obtiene un permiso (bloqueando si no hay ninguno disponible). release() devuelve un permiso. Si el semáforo tiene N permisos, como máximo N hilos pueden estar en la sección protegida simultáneamente.

En Big Data / stress testing la aplicación es directa: limitar la tasa de peticiones concurrentes. Si envías 1000 hilos contra Kafka sin límite, puedes matar al broker antes de poder medir nada útil. Con Semaphore(50) mantienes exactamente 50 peticiones en vuelo en todo momento. 

Semaphore(3):  [permiso][permiso][permiso]

  T1 acquire() ─► [■][permiso][permiso]   T1 dentro
  T2 acquire() ─► [■][■][permiso]          T2 dentro
  T3 acquire() ─► [■][■][■]               T3 dentro

  T4 acquire() ─► BLOQUEADO (0 permisos)
  T5 acquire() ─► BLOQUEADO

  T1 release() ─► [permiso][■][■]         T4 entra
  T3 release() ─► [permiso][permiso][■]   T5 entra

Ejemplo — throttling de peticiones a Kafka

KafkaThrottle.java 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class KafkaThrottle {

    static final int TOTAL_MSGS  = 10_000;
    static final int MAX_INFLY   = 100;    // máximo concurrente
    static final AtomicLong enviados  = new AtomicLong(0);
    static final AtomicLong errores   = new AtomicLong(0);

    public static void main(String[] args) throws InterruptedException {
        Semaphore inFlight = new Semaphore(MAX_INFLY);
        ExecutorService pool = Executors.newFixedThreadPool(MAX_INFLY * 2);

        long t0 = System.currentTimeMillis();

        for (int i = 0; i < TOTAL_MSGS; i++) {
            inFlight.acquire();  // bloquea si ya hay MAX_INFLY en vuelo
            final int msgId = i;

            pool.submit(() -> {
                try {
                    // Simula envío a Kafka (latencia variable)
                    long latencia = (long)(Math.random() * 10 + 1);
                    Thread.sleep(latencia);

                    if (Math.random() < 0.01) {  // 1% de error
                        errores.incrementAndGet();
                    } else {
                        enviados.incrementAndGet();
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    inFlight.release();  // siempre liberar el permiso
                }
            });
        }

        pool.shutdown();
        pool.awaitTermination(60, TimeUnit.SECONDS);

        long elapsed = System.currentTimeMillis() - t0;
        System.out.printf("Enviados: %d | Errores: %d | Throughput: %.0f msg/s%n",
            enviados.get(), errores.get(),
            TOTAL_MSGS / (elapsed / 1000.0));
    }
}
WARN Siempre pon inFlight.release() en un bloque finally. Si el permiso no se devuelve por una excepción, el semáforo se agota progresivamente hasta que no hay permisos y el sistema se congela — un deadlock silencioso muy difícil de diagnosticar bajo carga.

Semaphore vs BlockingQueue — cuándo usar cuál

Criterio Semaphore BlockingQueue
¿Qué controla? Número de hilos en sección Cola de elementos con capacidad
Backpressure No (bloquea al productor) Sí (productor espera si llena)
Caso de uso Limitar concurrencia de operaciones Comunicar datos entre hilos
Stress testing Throttling de conexiones Buffer de mensajes entre productores/consumidores

4 · CompletableFuture — pipelines reactivos asíncronos

Qué es y por qué importa

CompletableFuture<T> (Java 8+) es la respuesta de Java a la programación reactiva. Un Future tradicional sólo tiene get(), que bloquea. CompletableFuture permite encadenar operaciones asíncronas sin bloquear ningún hilo: cuando el resultado esté disponible, se ejecuta la siguiente etapa del pipeline automáticamente en un pool de hilos (por defecto ForkJoinPool.commonPool()).

Ejemplo base — pipeline de petición HTTP async

PipelineHTTP.java 
import java.net.URI;
import java.net.http.*;
import java.time.Duration;
import java.util.concurrent.*;

public class PipelineHTTP {

    static final HttpClient http = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(5))
        .build();

    public static CompletableFuture<String> fetchAsync(String url) {
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create(url))
            .timeout(Duration.ofSeconds(10))
            .GET()
            .build();

        return http.sendAsync(request, HttpResponse.BodyHandlers.ofString())
            .thenApply(response -> {
                if (response.statusCode() != 200) {
                    throw new RuntimeException("HTTP " + response.statusCode());
                }
                return response.body();
            })
            .thenApply(body -> {
                // Transformar: extraer longitud como métrica
                return "OK len=" + body.length();
            })
            .exceptionally(ex -> "ERROR: " + ex.getMessage());
    }

    public static void main(String[] args) throws Exception {
        // 5 peticiones paralelas sin bloquear ningún hilo
        CompletableFuture<String>[] futures = new CompletableFuture[5];
        String[] urls = {
            "https://httpbin.org/get",
            "https://httpbin.org/delay/1",
            "https://httpbin.org/status/500",
            "https://httpbin.org/get?q=test",
            "https://httpbin.org/headers"
        };

        for (int i = 0; i < urls.length; i++) {
            futures[i] = fetchAsync(urls[i]);
        }

        // allOf espera a todos sin bloquear un hilo dedicado
        CompletableFuture.allOf(futures)
            .thenRun(() -> {
                for (int i = 0; i < futures.length; i++) {
                    System.out.printf("[%d] %s%n", i, futures[i].join());
                }
            })
            .get();  // único get() bloqueante, al final del main
    }
}

Ejemplo real — medición de latencias en paralelo para stress report

LatencyCollector.java 
import java.util.*;
import java.util.concurrent.*;
import java.util.stream.*;

public class LatencyCollector {

    static long simulatePeticion() throws InterruptedException {
        long t0 = System.nanoTime();
        Thread.sleep((long)(Math.random() * 100 + 5));  // 5-105 ms
        return (System.nanoTime() - t0) / 1_000_000L;   // en ms
    }

    public static void main(String[] args) throws Exception {
        int N = 200;
        ExecutorService pool = Executors.newFixedThreadPool(50);

        List<CompletableFuture<Long>> futures = IntStream.range(0, N)
            .mapToObj(i -> CompletableFuture.supplyAsync(() -> {
                try {
                    return simulatePeticion();
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    return -1L;
                }
            }, pool))
            .collect(Collectors.toList());

        // Esperar a todos y recoger latencias
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();

        List<Long> latencias = futures.stream()
            .map(CompletableFuture::join)
            .filter(l -> l >= 0)
            .sorted()
            .collect(Collectors.toList());

        // Calcular percentiles
        int n = latencias.size();
        System.out.printf("N=%d peticiones%n", n);
        System.out.printf("p50 = %d ms%n", latencias.get((int)(n * 0.50)));
        System.out.printf("p95 = %d ms%n", latencias.get((int)(n * 0.95)));
        System.out.printf("p99 = %d ms%n", latencias.get((int)(n * 0.99)));
        System.out.printf("max = %d ms%n", latencias.get(n - 1));

        pool.shutdown();
    }
}
INFO supplyAsync(supplier, pool) usa tu pool propio. Sin el segundo argumento usa ForkJoinPool.commonPool(), que tiene tantos hilos como cores. Para trabajo IO-bound (peticiones HTTP, Kafka) necesitas un pool dedicado más grande.

thenCompose — encadenar futuros dependientes

thenApply transforma el valor. thenCompose lo usa para iniciar otra operación asíncrona. La diferencia es la misma que entre map y flatMap en streams.

Compose — autenticación + petición 
CompletableFuture<String> flujo =
    CompletableFuture.supplyAsync(() -> obtenerToken())       // devuelve String
        .thenCompose(token ->                                  // usa ese String para
            CompletableFuture.supplyAsync(() ->               // otro CF
                hacerPeticionConToken(token)))
        .thenApply(respuesta -> parsearJSON(respuesta))
        .exceptionally(ex -> {
            System.err.println("Error en flujo: " + ex.getMessage());
            return null;
        });

String resultado = flujo.get(10, TimeUnit.SECONDS);

5 · Fork/Join Framework — divide y vencerás a escala JVM

Arquitectura

El ForkJoinPool (Java 7+) implementa work-stealing: cada hilo tiene su propia cola de tareas. Cuando un hilo se queda sin trabajo, roba tareas de la cola de otro hilo. Esto maximiza la utilización de los cores sin contención en una cola central.

El modelo de programación es recursivo: ForkJoinTask se divide en subtareas más pequeñas (fork()) y espera sus resultados (join()). Hay dos sabores:

RecursiveTask<T>

Computa un resultado. Equivalente a Callable en el mundo Fork/Join. Úsalo cuando necesitas un valor de retorno.

RecursiveAction

No devuelve resultado. Equivalente a Runnable. Úsalo para mutaciones en colecciones o arrays compartidos.

  ForkJoinPool (N hilos = cores disponibles)
  │
  └─► tarea(0..1M)
       │
       ├─► fork: tarea(0..500K)           ├─► fork: tarea(500K..1M)
       │    ├─► fork: tarea(0..250K)      │    ├─► fork: tarea(500K..750K)
       │    │    [resultado: 125K]         │    │    [resultado: 125K]
       │    └─► fork: tarea(250K..500K)   │    └─► fork: tarea(750K..1M)
       │         [resultado: 125K]         │         [resultado: 125K]
       │    join() → 250K                  │    join() → 250K
       └─────────────────── join() ────────┘
                                 ↓
                          [resultado: 500K]

Ejemplo base — suma paralela de array grande

SumaParalela.java 
import java.util.concurrent.*;

public class SumaParalela extends RecursiveTask<Long> {

    static final int UMBRAL = 10_000;  // procesar secuencialmente por debajo de este tamaño

    private final long[] datos;
    private final int desde;
    private final int hasta;

    public SumaParalela(long[] datos, int desde, int hasta) {
        this.datos = datos;
        this.desde = desde;
        this.hasta = hasta;
    }

    @Override
    protected Long compute() {
        int longitud = hasta - desde;

        // Caso base: lo suficientemente pequeño para procesar directamente
        if (longitud <= UMBRAL) {
            long suma = 0;
            for (int i = desde; i < hasta; i++) {
                suma += datos[i];
            }
            return suma;
        }

        // Caso recursivo: dividir
        int medio = desde + longitud / 2;
        SumaParalela izq = new SumaParalela(datos, desde, medio);
        SumaParalela der = new SumaParalela(datos, medio, hasta);

        izq.fork();              // lanza izq en otro hilo
        long sumaDer = der.compute();  // computa der en este hilo
        long sumaIzq = izq.join();    // espera el resultado de izq

        return sumaIzq + sumaDer;
    }

    public static void main(String[] args) {
        int N = 10_000_000;
        long[] datos = new long[N];
        for (int i = 0; i < N; i++) datos[i] = i + 1L;
        long esperado = (long)N * (N + 1) / 2;

        ForkJoinPool pool = new ForkJoinPool();

        long t0 = System.currentTimeMillis();
        long resultado = pool.invoke(new SumaParalela(datos, 0, N));
        long elapsed = System.currentTimeMillis() - t0;

        System.out.printf("Resultado: %d (esperado: %d) — %s%n",
            resultado, esperado, resultado == esperado ? "CORRECTO" : "ERROR");
        System.out.printf("Tiempo: %d ms con %d cores%n",
            elapsed, Runtime.getRuntime().availableProcessors());

        pool.shutdown();
    }
}

Ejemplo real — procesamiento paralelo masivo de logs de Kafka

Escenario: tienes 50 millones de eventos de log de un clúster Kafka exportados a un array (o lista). Necesitas calcular la tasa de error por servicio. Fork/Join divide el array en bloques y cada bloque construye su propio mapa parcial. La reducción final combina los mapas.

LogAnalyzer.java 
import java.util.*;
import java.util.concurrent.*;

// Evento de log simplificado
record LogEvento(String servicio, boolean esError) {}

public class LogAnalyzer extends RecursiveTask<Map<String, long[]>> {

    static final int UMBRAL = 50_000;
    private final List<LogEvento> eventos;
    private final int desde;
    private final int hasta;

    public LogAnalyzer(List<LogEvento> eventos, int desde, int hasta) {
        this.eventos = eventos;
        this.desde   = desde;
        this.hasta   = hasta;
    }

    @Override
    protected Map<String, long[]> compute() {
        if (hasta - desde <= UMBRAL) {
            return procesarSegmento();
        }

        int medio = desde + (hasta - desde) / 2;
        LogAnalyzer izq = new LogAnalyzer(eventos, desde, medio);
        LogAnalyzer der = new LogAnalyzer(eventos, medio, hasta);

        izq.fork();
        Map<String, long[]> mapaDer = der.compute();
        Map<String, long[]> mapaIzq = izq.join();

        return combinar(mapaIzq, mapaDer);
    }

    // [total, errores] por servicio
    private Map<String, long[]> procesarSegmento() {
        Map<String, long[]> mapa = new HashMap<>();
        for (int i = desde; i < hasta; i++) {
            LogEvento e = eventos.get(i);
            long[] contadores = mapa.computeIfAbsent(e.servicio(), k -> new long[]{0, 0});
            contadores[0]++;
            if (e.esError()) contadores[1]++;
        }
        return mapa;
    }

    private Map<String, long[]> combinar(Map<String, long[]> a, Map<String, long[]> b) {
        Map<String, long[]> resultado = new HashMap<>(a);
        b.forEach((k, v) -> resultado.merge(k, v,
            (va, vb) -> new long[]{va[0] + vb[0], va[1] + vb[1]}));
        return resultado;
    }

    public static void main(String[] args) {
        String[] servicios = {"api-gateway", "kafka-consumer", "spark-job", "hdfs-writer"};
        int N = 5_000_000;
        List<LogEvento> eventos = new ArrayList<>(N);
        Random rand = new Random();
        for (int i = 0; i < N; i++) {
            eventos.add(new LogEvento(
                servicios[rand.nextInt(servicios.length)],
                rand.nextDouble() < 0.05  // 5% error rate
            ));
        }

        ForkJoinPool pool = ForkJoinPool.commonPool();
        long t0 = System.currentTimeMillis();
        Map<String, long[]> resultado = pool.invoke(new LogAnalyzer(eventos, 0, N));
        System.out.printf("Análisis de %dM eventos en %d ms%n", N / 1_000_000,
            System.currentTimeMillis() - t0);

        resultado.forEach((svc, c) ->
            System.out.printf("%-20s  total=%7d  errores=%6d  tasa=%.1f%%%n",
                svc, c[0], c[1], 100.0 * c[1] / c[0]));
    }
}

Caso de fallo — umbral demasiado pequeño

FAIL Si el umbral de recursión es demasiado pequeño (ej: 10 en lugar de 10.000), el overhead de crear tareas Fork/Join supera el beneficio del paralelismo. Con N=10M y umbral=10 se crean 1 millón de tareas — el scheduler se satura. La regla general: umbral entre 1.000 y 100.000 según el coste de la operación por elemento. Mide siempre.

6 · Integración — stress tester completo con todo el arsenal JUC

Combinamos todo lo visto (CountDownLatch + Semaphore + CompletableFuture) en un stress tester reutilizable. Este es el patrón base sobre el que construir herramientas específicas para Kafka, Spark REST, HDFS, etc.

JucStressTester.java 
import java.util.*;
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;
import java.util.stream.*;

public class JucStressTester {

    // ── Configuración ──────────────────────────────────────────
    static final int WORKERS       = 200;   // hilos totales
    static final int MAX_INFLIGHT  = 50;    // Semaphore: concurrencia máxima
    static final int TOTAL_OPS     = 5_000; // operaciones totales
    static final int WARMUP_OPS    = 500;   // descartar métricas de calentamiento

    // ── Métricas ───────────────────────────────────────────────
    static final List<Long> latencias = Collections.synchronizedList(new ArrayList<>());
    static final AtomicLong exitos  = new AtomicLong(0);
    static final AtomicLong fallos  = new AtomicLong(0);

    // ── Operación a estresar (sustituir por Kafka/HTTP/Spark) ──
    static long operacion(int id) throws InterruptedException {
        long t0 = System.nanoTime();
        Thread.sleep((long)(Math.random() * 80 + 10));  // 10-90 ms
        if (Math.random() < 0.02) throw new RuntimeException("Error simulado #" + id);
        return (System.nanoTime() - t0) / 1_000_000L;
    }

    public static void main(String[] args) throws Exception {
        ExecutorService pool   = Executors.newFixedThreadPool(WORKERS);
        Semaphore inFlight     = new Semaphore(MAX_INFLIGHT);
        CountDownLatch listos  = new CountDownLatch(WORKERS > TOTAL_OPS ? TOTAL_OPS : WORKERS);
        CountDownLatch disparo = new CountDownLatch(1);

        System.out.printf("Stress tester: %d ops, max %d concurrentes%n",
            TOTAL_OPS, MAX_INFLIGHT);

        List<CompletableFuture<Void>> futures = new ArrayList<>();
        final AtomicInteger warmupRestante = new AtomicInteger(WARMUP_OPS);

        for (int i = 0; i < TOTAL_OPS; i++) {
            final int opId = i;
            CompletableFuture<Void> f = CompletableFuture.runAsync(() -> {
                try {
                    if (opId < listos.getCount()) listos.countDown();
                    disparo.await();          // espera señal de disparo
                    inFlight.acquire();       // limita concurrencia

                    long lat = operacion(opId);

                    // descartar warmup
                    if (warmupRestante.decrementAndGet() < 0) {
                        latencias.add(lat);
                        exitos.incrementAndGet();
                    }
                } catch (Exception e) {
                    fallos.incrementAndGet();
                } finally {
                    inFlight.release();
                }
            }, pool);
            futures.add(f);
        }

        listos.await();  // esperar arranque de workers iniciales
        System.out.println("DISPARO");
        long t0 = System.currentTimeMillis();
        disparo.countDown();

        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).get();
        long elapsed = System.currentTimeMillis() - t0;

        // ── Reporte ────────────────────────────────────────────
        List<Long> sorted = latencias.stream().sorted().collect(Collectors.toList());
        int n = sorted.size();

        System.out.println("\n══════════════ REPORTE ══════════════");
        System.out.printf("Duración total : %.2f s%n", elapsed / 1000.0);
        System.out.printf("Throughput     : %.0f ops/s%n", TOTAL_OPS / (elapsed / 1000.0));
        System.out.printf("Exitos         : %d%n", exitos.get());
        System.out.printf("Fallos         : %d (%.1f%%)%n", fallos.get(),
            100.0 * fallos.get() / TOTAL_OPS);
        if (n > 0) {
            System.out.printf("p50            : %d ms%n", sorted.get((int)(n * 0.50)));
            System.out.printf("p95            : %d ms%n", sorted.get((int)(n * 0.95)));
            System.out.printf("p99            : %d ms%n", sorted.get((int)(n * 0.99)));
            System.out.printf("max            : %d ms%n", sorted.get(n - 1));
        }
        System.out.println("═════════════════════════════════════");

        pool.shutdown();
    }
}

Resumen — cuándo usar cada primitiva JUC

Primitiva Patrón En stress testing Limitación
CountDownLatch Esperar a N eventos; señal de arranque Disparo simultáneo de workers No reutilizable
CyclicBarrier Sincronización por fases Warm-up → Ramp → Pico → Cool BrokenBarrierException si un hilo falla
Semaphore Limitar concurrencia Throttling de peticiones en vuelo No FIFO por defecto
CompletableFuture Pipeline reactivo sin bloqueos Recolección de métricas async Requiere pool IO separado para trabajo pesado
ForkJoinPool Divide y vencerás CPU-bound Análisis post-test de resultados masivos No eficiente para IO-bound (hilos bloqueados)
