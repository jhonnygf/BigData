Threads en Java: sin GIL, con paralelismo real

Thread, Runnable, ExecutorService. synchronized, volatile, AtomicInteger. El productor-consumidor con BlockingQueue. Por qué Java escala donde Python tiene límites.

Java y el GIL: la diferencia fundamental

Java no tiene GIL. La JVM ejecuta múltiples threads en paralelo real en núcleos distintos. No hay mecanismo global que serialice la ejecución: cada thread en Java puede ejecutar bytecode JVM al mismo tiempo en su propio núcleo. Esta es la razón por la que herramientas como Gatling, JMeter, o Kafka brokers están escritas en Java/Scala: pueden saturar muchos núcleos desde un solo proceso sin las limitaciones de Python.

Python threading
  • GIL: solo un thread ejecuta bytecode a la vez
  • IO-bound: threading escala (GIL se libera en I/O)
  • CPU-bound: threading NO escala (GIL bloquea)
  • Solución CPU: multiprocessing (procesos separados)
Java threading
  • Sin GIL: múltiples threads ejecutan en paralelo real
  • IO-bound: threading escala perfectamente
  • CPU-bound: threading escala (usa múltiples núcleos)
  • Un solo proceso puede saturar todos los núcleos
WARN
Sin GIL = más responsabilidad. En Python, el GIL da cierta protección implícita (operaciones simples sobre objetos Python son atómicas). En Java, no hay ninguna protección implícita. Cualquier dato compartido entre threads requiere sincronización explícita con synchronized, volatile, o clases de java.util.concurrent.

Crear threads: Thread y Runnable

Opción 1: extender Thread

java — Thread básico extendiendo la clase 
public class MiThread extends Thread {
    private final String nombre;
    private final int duracionMs;

    public MiThread(String nombre, int duracionMs) {
        this.nombre = nombre;
        this.duracionMs = duracionMs;
    }

    @Override
    public void run() {
        System.out.println("[" + nombre + "] Arrancando en thread: " + Thread.currentThread().getName());
        try {
            Thread.sleep(duracionMs);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // restaurar flag de interrupción
            System.out.println("[" + nombre + "] Interrumpido");
            return;
        }
        System.out.println("[" + nombre + "] Terminado");
    }

    public static void main(String[] args) throws InterruptedException {
        MiThread t1 = new MiThread("A", 1000);
        MiThread t2 = new MiThread("B", 500);
        MiThread t3 = new MiThread("C", 1500);

        long inicio = System.currentTimeMillis();
        t1.start();
        t2.start();
        t3.start();

        t1.join(); // esperar a que termine
        t2.join();
        t3.join();

        long elapsed = System.currentTimeMillis() - inicio;
        System.out.println("Todos terminados en " + elapsed + "ms (secuencial sería ~3000ms)");
    }
}

Opción 2: Runnable con lambda (forma moderna)

java — Runnable con lambda (Java 8+) 
public class EjemploRunnable {
    public static void main(String[] args) throws InterruptedException {
        // Runnable como lambda: más limpio que extender Thread
        Runnable tarea = () -> {
            System.out.println("Thread: " + Thread.currentThread().getName());
            try { Thread.sleep(500); } catch (InterruptedException e) { return; }
            System.out.println("Tarea terminada");
        };

        Thread t = new Thread(tarea, "mi-thread-1");
        t.start();
        t.join();

        // Opción aún más compacta con lambda inline
        Thread t2 = new Thread(() -> System.out.println("Thread inline"), "inline");
        t2.start();
        t2.join();
    }
}

ExecutorService: el pool de threads de Java

Igual que en Python con ThreadPoolExecutor, en Java la forma correcta de gestionar múltiples threads es con un ExecutorService. Crea el pool una vez, reutiliza los threads para muchas tareas, y lo cierra ordenadamente al final.

java — ExecutorService: pool de threads para Big Data 
import java.util.concurrent.*;
import java.util.*;

public class EjemploPool {

    // Simula una petición HTTP a un sistema Big Data
    static long peticionHTTP(int id) throws Exception {
        long t0 = System.currentTimeMillis();
        Thread.sleep(100 + (int)(Math.random() * 200)); // simula latencia de red
        return System.currentTimeMillis() - t0;
    }

    public static void main(String[] args) throws InterruptedException {
        int N_PETICIONES = 100;
        int N_THREADS = 20;

        ExecutorService pool = Executors.newFixedThreadPool(N_THREADS);
        List> futures = new ArrayList<>();

        long inicio = System.currentTimeMillis();

        // Enviar todas las tareas al pool
        for (int i = 0; i < N_PETICIONES; i++) {
            final int id = i;
            Future future = pool.submit(() -> peticionHTTP(id));
            futures.add(future);
        }

        // Recoger resultados
        List latencias = new ArrayList<>();
        int errores = 0;
        for (Future f : futures) {
            try {
                latencias.add(f.get()); // bloquea hasta que el future esté listo
            } catch (ExecutionException e) {
                errores++;
            }
        }

        pool.shutdown();
        pool.awaitTermination(30, TimeUnit.SECONDS);

        long elapsed = System.currentTimeMillis() - inicio;
        Collections.sort(latencias);

        System.out.printf("Peticiones:    %d%n", N_PETICIONES);
        System.out.printf("Errores:       %d%n", errores);
        System.out.printf("Duración:      %dms%n", elapsed);
        System.out.printf("RPS:           %.1f%n", N_PETICIONES * 1000.0 / elapsed);
        System.out.printf("p50:           %dms%n", latencias.get(latencias.size() / 2));
        System.out.printf("p95:           %dms%n", latencias.get((int)(latencias.size() * 0.95)));
        System.out.printf("p99:           %dms%n", latencias.get((int)(latencias.size() * 0.99)));
    }
}

synchronized: exclusión mutua en Java

synchronized en Java es el equivalente de threading.Lock en Python. Garantiza que solo un thread ejecuta el bloque marcado a la vez. Puede aplicarse a un método entero o a un bloque específico.

java — race condition y corrección con synchronized 
import java.util.concurrent.*;

public class ContadorRace {

    // SIN sincronización: race condition
    static int contadorInseguro = 0;

    static void incrementarInseguro() {
        for (int i = 0; i < 100_000; i++) {
            contadorInseguro++; // NO atómico: read + add + write
        }
    }

    // CON synchronized: thread-safe
    static int contadorSeguro = 0;

    static synchronized void incrementarSeguro() {
        for (int i = 0; i < 100_000; i++) {
            contadorSeguro++;
        }
    }

    public static void main(String[] args) throws InterruptedException {
        // Test sin sincronización
        ExecutorService pool = Executors.newFixedThreadPool(5);
        for (int i = 0; i < 5; i++) {
            pool.submit(ContadorRace::incrementarInseguro);
        }
        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);
        System.out.println("Sin sync  → Esperado: 500,000 | Obtenido: " + contadorInseguro);

        // Test con synchronized
        pool = Executors.newFixedThreadPool(5);
        for (int i = 0; i < 5; i++) {
            pool.submit(ContadorRace::incrementarSeguro);
        }
        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);
        System.out.println("Con sync  → Esperado: 500,000 | Obtenido: " + contadorSeguro);
    }
}

Output:

output
Sin sync  → Esperado: 500,000 | Obtenido: 387,234  ← incorrecto, varía
Con sync  → Esperado: 500,000 | Obtenido: 500,000  ← siempre correcto

AtomicInteger: más rápido que synchronized para contadores

AtomicInteger usa instrucciones CAS (Compare-And-Swap) del procesador: operaciones atómicas a nivel de hardware, sin locks. Para contadores simples es mucho más eficiente que synchronized.

java — AtomicInteger para métricas de alta frecuencia 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class MetricasAtomicas {
    // AtomicInteger: sin lock, operaciones atómicas por hardware
    static final AtomicInteger totalOK    = new AtomicInteger(0);
    static final AtomicInteger totalFail  = new AtomicInteger(0);
    static final AtomicLong    totalMs    = new AtomicLong(0);

    static void simularPeticion() {
        long t0 = System.currentTimeMillis();
        try {
            Thread.sleep((long)(Math.random() * 200 + 50));
            if (Math.random() < 0.05) throw new Exception("Error simulado");
            totalOK.incrementAndGet();                             // atómico
            totalMs.addAndGet(System.currentTimeMillis() - t0);   // atómico
        } catch (Exception e) {
            totalFail.incrementAndGet();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(50);
        for (int i = 0; i < 1000; i++) {
            pool.submit(MetricasAtomicas::simularPeticion);
        }
        pool.shutdown();
        pool.awaitTermination(60, TimeUnit.SECONDS);

        int ok   = totalOK.get();
        int fail = totalFail.get();
        long ms  = totalMs.get();

        System.out.printf("OK:    %d | Fail: %d | Fail%%: %.1f%n", ok, fail, fail*100.0/(ok+fail));
        System.out.printf("Latencia media (OK): %.0fms%n", ok > 0 ? ms * 1.0 / ok : 0);
    }
}

BlockingQueue: productor-consumidor thread-safe

BlockingQueue en Java es el equivalente de queue.Queue en Python. put() bloquea si la cola está llena; take() bloquea si está vacía. Es la base del patrón productor-consumidor en Java.

java — productor-consumidor con LinkedBlockingQueue 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class ProductorConsumidor {

    record Mensaje(int id, String datos) {}
    static final Mensaje FIN = new Mensaje(-1, "FIN"); // señal de parada

    static void productor(BlockingQueue cola, int n) {
        for (int i = 0; i < n; i++) {
            try {
                cola.put(new Mensaje(i, "datos_" + i));
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return;
            }
        }
        try {
            cola.put(FIN); // señal de fin
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    static AtomicInteger procesados = new AtomicInteger(0);

    static void consumidor(BlockingQueue cola, String nombre) {
        while (true) {
            try {
                Mensaje m = cola.take();
                if (m == FIN) {
                    cola.put(FIN); // reinyectar para otros consumidores
                    break;
                }
                Thread.sleep(10); // simula procesamiento
                procesados.incrementAndGet();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return;
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue cola = new LinkedBlockingQueue<>(100); // buffer de 100
        ExecutorService pool = Executors.newCachedThreadPool();

        long t0 = System.currentTimeMillis();
        pool.submit(() -> productor(cola, 500));
        for (int i = 0; i < 4; i++) {
            final String nombre = "C" + i;
            pool.submit(() -> consumidor(cola, nombre));
        }

        pool.shutdown();
        pool.awaitTermination(60, TimeUnit.SECONDS);
        System.out.printf("Procesados: %d en %dms%n", procesados.get(),
                          System.currentTimeMillis() - t0);
    }
}

volatile: visibilidad entre threads

volatile garantiza que el valor de una variable siempre se lea desde la memoria principal y no desde la caché de CPU del thread. Sin volatile, un thread puede ver un valor obsoleto de una variable que otro thread ha modificado.

java — volatile para stop flag compartido 
public class StopFlag {

    // Sin volatile: el JIT puede cachear el valor en el registro del thread
    // y el thread puede no ver la actualización del thread principal
    static volatile boolean parar = false;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            int iteraciones = 0;
            while (!parar) {  // sin volatile, podría ejecutar para siempre
                iteraciones++;
            }
            System.out.println("Worker detenido tras " + iteraciones + " iteraciones");
        });

        worker.start();
        Thread.sleep(100); // dejar que el worker ejecute un rato
        parar = true;      // visible inmediatamente gracias a volatile
        worker.join();
        System.out.println("Main: worker terminado");
    }
}
INFO
volatile no es suficiente para operaciones compuestas. volatile garantiza visibilidad pero no atomicidad. Para un contador compartido, volatile int contador no es thread-safe porque contador++ es tres operaciones. Usa AtomicInteger para eso.
